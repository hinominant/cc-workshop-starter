# Quality Gate

実装完了後の品質チェック〜コミットまでの一連のプロセスを実行する。
**3フェーズ検証（標準テスト×2 → 視点違いテスト×2 → 外部監査×1）+ Guardrail 判定 + commit まで一括実行。**

**省略・入れ替え禁止。**

## 実行手順

### Phase A: 標準テスト ×2（安定性確認）

同じテストスイートを2回実行して不安定テストを検出する。
1回目で失敗したら**ここで止めて修正する**（Phase B に進まない）。

テストランナーはプロジェクトに応じて選択:
- `npm test` / `npx vitest` / `npx jest`
- `python -m pytest` / `go test ./...`
- テストがない場合: `bash -n` (シェル構文) + lint のみ

1回目通過 → 2回目を実行。2回とも PASS したら Phase B へ。

---

### Phase B: 視点違いテスト ×2（セキュリティ・バイパス視点）

同じテストスイートを「**攻撃者・バイパス試行の視点**」で2回評価する。

**B-1: テストカバレッジギャップ確認**

1. テストディレクトリの既存テストケース一覧を読む
2. 変更したファイルに対応するテストが存在するか確認
3. 未テストの変更がある場合 → テストを追加してから Phase A に戻る

**B-2: 設定ファイル整合性確認**

1. コードの変更がドキュメント（CLAUDE.md, README, docs/）に反映されているか
2. 設定ファイル（.env.example, config 等）が実装と一致しているか
3. 問題が見つかった場合 → 修正して Phase A に戻る

---

### Phase C: 外部監査（Sentinel）×1

**前の会話コンテキストをリセットした独立エージェントとして**、以下の3観点でコードレビューを実施する。

> 重要: このステップは「今まで何を作っていたか」を意図的に忘れ、コードを初めて見る第三者として評価する。

**C-1: セキュリティ（S）**
- ハードコードされたシークレット・API キーがないか
- 外部入力のバリデーション漏れ、インジェクション脆弱性

**C-2: 設計一貫性（D）**
- コードスタイルの一貫性
- 命名規則の統一
- 不要な複雑性がないか

**C-3: エッジケース（E）**
- 空入力、null、undefined での動作
- 境界値（0, 負数, 最大値）
- エラーハンドリングの漏れ

**問題が見つかった場合:**
1. 問題を列挙して報告
2. 修正を実施
3. Phase A からやり直す

**問題がなければ:** 「Sentinel: PASS」と宣言して Step 1 へ。

---

### Step 1: Guardrail レベル判定

| Level | 条件 | 対応 |
|-------|------|------|
| **L1** | ドキュメントのみ、軽微な修正 | 簡易確認で通過 |
| **L2** | 通常の機能開発、バグ修正、テスト追加 | 全 Phase 実施済みなら通過 |
| **L3** | 大規模変更、破壊的変更 | Phase C の厳格実施 + 影響分析 |
| **L4** | セキュリティ脆弱性、データ整合性リスク | 即時停止 → ユーザーに報告 |

「Guardrail: L{N} ({理由})」と明示。

---

### Step 2: git commit + push

全 Phase 通過後にコミット・プッシュする。

- `git status` で変更内容を確認
- `git diff` で差分を確認
- `git log --oneline -5` でコミットメッセージスタイルを確認
- 適切なコミットメッセージで commit + push

---

### Step 3: ARIS パターン記録（該当時のみ）

**成功パターン記録条件** → `/log-success` を実行:
- フェーズやマイルストーンが完了した
- 想定より大幅に効率的な手法を発見・適用できた

**失敗パターン記録条件** → `/log-failure` を実行:
- 途中で設計ミス・判断ミスがあり修正した
- Phase A/B/C でやり直しが発生した

記録対象がなければスキップ。

---

### Step 4: 完了報告

ユーザーに以下を簡潔に報告する:

- Phase A: テスト結果（pass/fail 件数）×2
- Phase B: カバレッジギャップ・設定整合性確認結果
- Phase C: Sentinel 監査結果
- Guardrail: L{N}
- コミットハッシュ

---

## ルール

- Phase A → B → C → Step 1 の順序は絶対。省略・スキップ禁止
- Phase A で失敗したら B/C に進まず修正する
- Phase C で問題が見つかった場合は Phase A に戻る
- L4 判定は即座にユーザーに報告して停止する
