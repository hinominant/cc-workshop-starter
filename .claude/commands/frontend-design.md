# Frontend Design Mode

「AIっぽい無難なUI」ではなく、意図を持ったビジュアルデザインを実現する。
実装前にデザイン意図を明確にし、根拠のある選択を積み重ねる。

## タスク

$ARGUMENTS

## デザイン原則

### 禁止事項（AIっぽいデザインの特徴）

以下を避ける:
- 均等な余白で並べただけのレイアウト
- 意味なく角丸にしたカード
- グレー一色の無個性な配色
- 全要素が同じサイズ・ウェイトのテキスト
- 装飾目的だけのグラデーション
- 何も伝えないストックフォト的アイコン

### タイポグラフィ

- **フォントサイズ階層**: 最低4段階（h1/h2/body/caption）を定義する。比率は 1.25 以上
- **行間（line-height）**: 本文は 1.5〜1.75、見出しは 1.1〜1.3
- **letter-spacing**: 見出しは -0.02em〜-0.01em で引き締める。本文は 0〜0.01em
- **フォントウェイト**: 最大3種類（regular/medium/bold）。多用しない
- **読みやすさ**: 1行の最大文字数は 60〜80文字（英語）/ 35〜45文字（日本語）

### 余白（Spacing）

- **基本グリッド**: 8px ベース（4px は微調整のみ）
- **コンポーネント内余白**: 16px / 24px
- **セクション間余白**: 48px / 64px / 96px（情報の区切りに応じて段階的に）
- **呼吸感**: 要素を詰め込まない。余白は「何もない」ではなく「意図的な間」
- **一貫性**: 同じ文脈の余白は同じ値を使う

### 配色

- **プライマリ**: ブランドの主色。CTA やアクティブ要素に使用
- **セカンダリ**: 補助色。プライマリと調和しつつ区別できる色
- **アクセント**: 注意喚起・ハイライト用。控えめに使う
- **ニュートラル**: 背景・テキスト・ボーダー用のグレースケール（最低5段階）
- **コントラスト比**: テキスト背景間は WCAG AA 準拠（4.5:1 以上）
- **セマンティック**: success(green) / warning(amber) / error(red) / info(blue)

### レイアウト

- **視線の流れ**: コンテンツページは F パターン、LP は Z パターンを意識
- **情報階層**: 最重要 → 補足 → 詳細 の3層構造
- **グリッドシステム**: 12カラムグリッド。モバイルは4カラム
- **最大幅**: コンテンツ領域は max-width を設定（1200px 目安）
- **アライメント**: 左揃え基本。中央揃えは意図的な場合のみ

### マイクロインタラクション

- **hover**: 色変化 + subtle な transform（translateY(-1px) 等）。150〜200ms
- **focus**: 明確な focus ring（outline）。キーボードナビゲーション対応
- **transition**: ease-out 基本。duration は 150〜300ms。500ms 以上は避ける
- **フィードバック**: ボタンクリック時は即座に視覚変化。ローディング中は skeleton/spinner
- **アニメーション**: 目的がある場合のみ。装飾目的の過剰なアニメーション禁止

### レスポンシブ

- **モバイルファースト**: min-width でブレイクポイント定義
- **ブレイクポイント**: sm(640px) / md(768px) / lg(1024px) / xl(1280px)
- **タッチターゲット**: 最小 44x44px
- **フォントサイズ**: モバイルでは本文 16px 以上（ズーム防止）
- **ナビゲーション**: モバイルではハンバーガーメニューまたはボトムナビ

## 実行プロセス

### Step 0: DESIGN.md の確認・生成

**実装前に必ず実行する。**

1. `.agents/DESIGN.md` が存在するか確認
2. **存在する場合**: 読み込んで Step 1 に進む
3. **存在しない場合**: 以下の優先順で DESIGN.md を生成する:
   - **Figma MCP 接続あり** → Figma からデザイントークンを抽出して DESIGN.md を生成
   - **参考 URL あり** → Browser Use CLI でサイトの CSS を解析して DESIGN.md を生成
   - **どちらもなし** → Step 1 のデザイン意図定義後に手動で DESIGN.md を作成

DESIGN.md の生成フォーマットは `skills/design-md.md` を参照。

### Step 1: デザインブリーフの構造化（3要素テンプレート）

**曖昧な指示のまま実装に入らない。** 以下の3要素を必ず明文化する:

```
Idea:    何を作るか。目的と理想のCTA（ユーザーに何をしてほしいか）
Theme:   どう見えるか。形容詞で雰囲気を定義する
Content: 何が必要か。セクション・要素・情報構造を列挙する
```

**記述例:**

```
Idea: マッチングサービスのLP。女性ユーザーが安心して登録できることがゴール。
      CTAは「無料で始める」ボタン。

Theme: 温かみがあり信頼感のある。柔らかいが甘すぎない。
       パステルではなく、落ち着いたアースカラー。
       余白を活かしたミニマルなレイアウト。

Content: ヒーローセクション（キャッチコピー + CTA）、
         3ステップの使い方説明、安全性への取り組み（本人確認・監視体制）、
         利用者の声（匿名・年代のみ表示）、料金プラン比較、
         よくある質問、フッターCTA。
```

**形容詞の選び方（Theme の品質を決める）:**

| 印象 | 使える形容詞 |
|------|-------------|
| 信頼・プロ | Clean, Modern, High-trust, Sharp, Executive-ready |
| 温かみ・親しみ | Warm, Inviting, Friendly, Approachable, Soft |
| 洗練・高級 | Elegant, Refined, Luxurious, Sophisticated, Premium |
| 活力・楽しさ | Vibrant, Energetic, Playful, Bold, Dynamic |
| 静寂・集中 | Minimalist, Focused, Calm, Zen, Serene |
| 重厚・力強さ | Dense, Industrial, Dark, Utilitarian, Robust |

**形容詞は2-3個に絞る。** 多すぎると方向性がブレる。

### Step 2: デザイントークンの決定

**DESIGN.md が存在する場合**: DESIGN.md の値をそのまま使用する。Step 3 に進む。

**存在しない場合**: Step 1 の Theme から逆算してトークンを決定し、DESIGN.md として保存する:

| 要素 | 決めること | Theme との対応 |
|------|-----------|---------------|
| カラーパレット | hex値 + セマンティック名 | 形容詞から色温度・彩度を導出 |
| タイポグラフィ | フォント + サイズスケール | 形容詞からウェイト・セリフ有無を導出 |
| スペーシング | 基本グリッド + スケール | "Airy"→広め、"Dense"→詰め |
| 角丸 | radius 値 | "Soft"→大きめ、"Sharp"→小さめ or 0 |
| シャドウ | 深度 | "Flat"→なし、"Elevated"→強め |
| トランジション | duration + easing | "Calm"→遅め、"Dynamic"→速め |

### Step 3: スクリーン単位で実装（1画面ずつ集中）

**一度に複数画面を実装しない。** 1画面ずつ完成させてから次に進む。

各スクリーンで:

1. **そのスクリーンの目的を1文で書く**（例: 「ユーザーに料金プランを比較させ、CTAをクリックさせる」）
2. **レイアウトの骨格**（HTML構造 + グリッド配置）
3. **タイポグラフィ階層**（最重要テキスト → 補足 → 詳細 の3層）
4. **カラー適用**（DESIGN.md のトークンのみ。独自 hex 禁止）
5. **インタラクション**（hover/focus/active — DESIGN.md の transition 値を使用）
6. **レスポンシブ**（モバイルファースト → タブレット → デスクトップ）
7. **そのスクリーンのスクリーンショット確認**

### Step 3.5: バリエーション検討（オプション）

重要なページ（LP ヒーロー、ダッシュボード、料金ページ等）では、**2-3 のバリエーション**を検討する:

- **レイアウトバリエーション**: 同じコンテンツで配置を変える（左右分割 vs 上下積み、カード vs リスト）
- **トーンバリエーション**: DESIGN.md の範囲内で色温度・コントラストを調整
- **情報量バリエーション**: 「もっとシンプル」vs「もっと詳細」

**最終的に1案に絞る。** バリエーションは検討のための手段であり、全部作るためではない。

### Step 4: デザイン品質レビュー

**6つのチェックポイント:**

| # | チェック | 具体的な確認方法 |
|---|---------|----------------|
| 1 | **視覚階層** | ページを3秒見て、最初に目に入るものが意図通りか |
| 2 | **一貫性** | 同じ役割の要素が同じスタイルか（ボタン色、テキストサイズ） |
| 3 | **余白の意図** | 詰まりすぎ・空きすぎがないか。余白に「理由」があるか |
| 4 | **コントラスト** | テキスト背景間 WCAG AA 準拠（4.5:1 以上） |
| 5 | **モバイル** | モバイルサイズで破綻しないか。タッチターゲット 44px 以上 |
| 6 | **AI臭さ** | 禁止事項リストに該当していないか |

**1つでも NG なら修正してから次のスクリーンに進む。**

### Step 5: コンポーネント仕様参照

実装するコンポーネントが以下に仕様定義されている場合、仕様に従って実装する:

- `_common/COMPONENT_SPEC.md` — コンポーネント仕様の標準テンプレート
- `agents/artisan/references/components/` — P1コンポーネント仕様（Button, Input, Select, Checkbox/Radio, Dialog, Table, Card）
- `agents/artisan/references/component-guidelines.md` — コンポーネント設計の原則・命名規則・ファイル構成

**パターン参照（複数コンポーネントの組み合わせ時）:**
- `agents/vision/references/patterns/` — デザインパターン仕様（DataTable, FormWizard, DeleteConfirmation, SearchFilter, SidebarLayout）
- `agents/vision/references/pattern-guidelines.md` — パターン選択の判断基準
- `agents/muse/references/ui-component-patterns.md` — UIコンポーネント100パターン辞書（15カテゴリ・用途・命名）

### Step 6: コンテンツ品質チェック

UIテキスト（ラベル・エラーメッセージ・確認ダイアログ等）を以下のガイドラインに照合する:

- `agents/palette/references/content-guidelines-ja.md` — 日本語UIコンテンツガイドライン（文体・ボタンラベル・エラーメッセージ・日付フォーマット等）
- `agents/palette/references/terminology-glossary.md` — UI用語辞書テンプレート
- `agents/palette/references/ux-writing-patterns.md` — 汎用UXライティングパターン
