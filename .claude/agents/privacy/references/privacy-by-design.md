# Privacy by Design 実装ガイド

> 最終更新: 2026-04-06
> 対象: privacy（主査）エージェント
> 用途: ソフトウェア開発におけるプライバシー保護の設計原則と実装パターン

---

## 1. Privacy by Design 7つの基本原則

| # | 原則 | 説明 | ソフトウェア開発での適用 |
|---|------|------|----------------------|
| 1 | **Proactive not Reactive** | 問題発生前に予防する | 設計段階でPIA実施、脅威モデリング |
| 2 | **Privacy as the Default** | デフォルトでプライバシーが最大限保護される | オプトイン方式、最小権限、デフォルト非公開 |
| 3 | **Privacy Embedded into Design** | 後付けではなく設計に組み込む | アーキテクチャレベルでのデータ保護設計 |
| 4 | **Full Functionality** | プライバシーと機能のトレードオフを避ける | 匿名化しても分析可能な設計 |
| 5 | **End-to-End Security** | ライフサイクル全体でデータを保護する | 取得→保存→利用→廃棄の全段階で保護 |
| 6 | **Visibility and Transparency** | 処理の透明性を確保する | 監査ログ、プライバシーダッシュボード |
| 7 | **Respect for User Privacy** | ユーザーの権利を最優先にする | セルフサービスの権利行使機能 |

## 2. データミニマイゼーションパターン

### 2.1 収集時のミニマイゼーション

| パターン | 説明 | 実装例 |
|---------|------|--------|
| 必要最小限の収集 | 利用目的に必要なデータのみ収集 | フォームの任意項目を必須にしない |
| 段階的収集 | サービス利用に応じて段階的にデータを取得 | 初回登録はメールのみ、利用時に追加情報を取得 |
| 代替データの利用 | 個人情報の代わりに非個人情報で代替 | 年齢の代わりに年代、住所の代わりに地域 |
| 集約データの利用 | 個人レベルではなく集約レベルで分析 | 個別行動履歴ではなくセグメント別傾向 |

### 2.2 保存時のミニマイゼーション

| パターン | 説明 | 実装例 |
|---------|------|--------|
| TTL（Time to Live） | データに有効期限を設定し自動削除 | セッションデータ30日、ログ1年 |
| データ分離 | 個人識別情報と利用データを物理的に分離 | 識別DBと行動DBの分離保管 |
| 参照の最小化 | 他テーブルとの結合を最小限にする | JOINに必要なキーのみを保持 |

## 3. 利用目的制限の強制パターン

### 3.1 技術的な利用目的制限

| パターン | 説明 | 実装 |
|---------|------|------|
| Purpose Binding | データに利用目的メタデータを紐付け | データベースに `purpose` カラムを追加。クエリ時にpurposeフィルタを強制 |
| Policy Enforcement Point | データアクセスを利用目的で制御 | APIゲートウェイで `X-Purpose` ヘッダーを必須化。不一致時は403 |
| Sticky Policy | データに利用条件を暗号的に結合 | 暗号化キーを利用目的と紐付け。目的外の復号を不可能にする |

### 3.2 組織的な利用目的制限

- [ ] データ利用申請プロセスが整備されているか
- [ ] 利用目的外のアクセスを検知する仕組みがあるか
- [ ] 目的外利用が発覚した場合の対応手順があるか

## 4. 同意管理の実装

### 4.1 同意管理システムの要件

| 要件 | 説明 | 実装 |
|------|------|------|
| 粒度の細かい同意 | 利用目的ごとに個別の同意/非同意 | チェックボックスの利用目的別分離 |
| 同意の記録 | 同意日時、同意内容、同意バージョンの記録 | `consent_records` テーブル（user_id, purpose, version, timestamp, action） |
| 同意の撤回 | いつでも撤回可能、撤回の容易性 | 設定画面からのワンクリック撤回 |
| 同意バージョン管理 | ポリシー変更時の再同意取得 | ポリシーバージョンと同意の紐付け |
| 未成年者の同意 | 法定代理人の同意取得 | 年齢確認 + 法定代理人のメール確認 |

### 4.2 同意取得のUI要件

- [ ] 同意は明示的な積極行為（チェックボックスのクリック等）で取得しているか（事前チェック済みはNG）
- [ ] 同意しない場合もサービスの基本機能が利用可能か（全同意を強制しない）
- [ ] 同意内容が平易な日本語で記載されているか
- [ ] 同意の撤回方法が同意取得と同等の容易さで提供されているか

## 5. データ保持・削除の自動化

### 5.1 データ保持ポリシーの設計

| データ種別 | 推奨保持期間 | 根拠 |
|-----------|------------|------|
| ユーザーアカウント情報 | サービス利用終了後6ヶ月（法的義務を除く） | 利用目的の達成 |
| 取引履歴 | 7年（商法・税法の帳簿保存義務） | 法令要件 |
| アクセスログ | 1年（セキュリティ監視目的） | 安全管理措置 |
| Cookie/セッション | 最長13ヶ月（GDPR推奨） | 国際基準 |
| 問い合わせ記録 | 対応完了後3年 | 時効を考慮 |
| AI学習用データ | 学習完了後速やかに削除（匿名加工後を除く） | 利用目的の達成 |

### 5.2 自動削除の実装パターン

```
[データ保持・削除フロー]

1. データ保持ポリシーの定義
   └── data_retention_policy テーブル: (data_type, retention_days, legal_basis)

2. 削除対象の自動検出
   └── 日次バッチ: retention_days を超えたレコードを検出

3. 削除前の確認
   ├── 法的保持義務のチェック（訴訟ホールド等）
   ├── 他テーブルへの参照の確認
   └── 削除対象リストの生成

4. 削除の実行
   ├── 論理削除（soft-delete）→ 一定期間後に物理削除
   └── 物理削除（hard-delete）+ 暗号化キーの破棄

5. 削除の記録
   └── deletion_log: (data_type, count, executed_at, method)
```

## 6. 仮名加工・匿名加工テクニック

### 6.1 仮名加工情報（第41条〜第46条）

| 手法 | 説明 | 適用場面 |
|------|------|---------|
| トークン化 | 識別子をランダムトークンに置換。マッピングテーブルで復元可能 | 社内分析、AI学習データ |
| ハッシュ化 | 識別子を一方向ハッシュに変換。ソルト必須 | 突合用途（ソルトで復元不可にする） |
| 一般化 | 詳細な値をカテゴリに変換（25歳→20代、港区→東京都） | 統計分析 |

### 6.2 匿名加工情報（第43条〜第46条）

| 手法 | 説明 | k-匿名性 |
|------|------|---------|
| k-匿名化 | 同一の属性組み合わせが少なくともk件存在するように加工 | k >= 5 推奨 |
| l-多様性 | 各グループ内で機微属性がl種類以上になるように加工 | l >= 3 推奨 |
| 差分プライバシー | ノイズを加えて個人の特定を数学的に不可能にする | epsilon = 1.0 以下推奨 |
| データ抑制 | 小さなグループのレコードを削除 | k未満のグループを除外 |

## 7. プライバシー影響評価（PIA）テンプレート

### 7.1 PIA実施基準

以下のいずれかに該当する場合、PIAを実施する:

- [ ] 新サービス・新機能で個人情報を新たに取得する
- [ ] 個人情報の利用目的を変更・追加する
- [ ] 新たな第三者提供・越境移転を開始する
- [ ] AI/機械学習で個人情報を利用する
- [ ] 大規模な個人情報の取扱いを開始する
- [ ] プロファイリング・自動意思決定を導入する

### 7.2 PIAテンプレート

```markdown
## プライバシー影響評価（PIA）

### 1. 基本情報
- 評価対象: [サービス名/機能名]
- 評価日: YYYY-MM-DD
- 評価者: [担当者名]
- 承認者: [個人情報保護責任者]

### 2. データフロー
- 取得する個人情報: [一覧]
- 取得方法: [直接取得/第三者提供/公開情報]
- 利用目的: [具体的な利用目的]
- 保存場所: [サーバー/クラウド/リージョン]
- 保存期間: [期間と根拠]
- 第三者提供: [あり/なし、提供先]
- 越境移転: [あり/なし、移転先国]

### 3. リスク評価

| 要件 | 準拠状況 | 備考 |
|------|---------|------|
| 利用目的の特定 | | |
| 同意取得 | | |
| 安全管理措置 | | |
| 第三者提供ルール | | |

| 本人の権利保障 | | |

### 5. 結論
- [ ] APPROVED: リスクが適切に軽減されている
- [ ] CONDITIONAL: 追加の軽減策を実施した上で承認
- [ ] REJECTED: リスクが許容範囲を超えている

### 6. 残存リスクと対応計画
| 残存リスク | 対応計画 | 期限 | 責任者 |
```

## 8. datashield へのハンドオフ時の補足

PbD設計のレビュー完了後、datashieldに以下の技術的検証を依頼する:

| 検証項目 | 目的 |
|---------|------|
| データミニマイゼーションの実装確認 | DB設計で不要なカラムが存在しないか |
| 同意管理の実装確認 | 同意の記録・撤回が正しく動作するか |
| 自動削除の実装確認 | TTLが正しく設定され、削除が実行されているか |
| 仮名加工/匿名加工の実装確認 | 加工処理が適切に実装されているか、復元リスクはないか |
| 暗号化・アクセス制御の実装確認 | PbD要件に基づく技術的保護措置の有効性 |
