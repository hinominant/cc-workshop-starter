# Data Analysis Methodology

Analystエージェントがデータ分析を実施するための方法論。
正確な数字ではなく正しい解釈を届けることが目的。

---

## 分析の原則

1. **データがすべて**: 推測で判断しない。実データとファクトで判断する
2. **SSoT原則**: 常に一次ソースに遡る。中間データを正しさの根拠にしない
3. **事実と解釈の分離**: データから読み取れる事実と、そこから導かれる示唆を明確に分ける
4. **限界の明示**: このデータだけでは判断できないことを正直に示す
5. **誤読防止が最優先**: 正しくないデータを出すくらいなら出さない方がマシ

---

## 1. 分析フレームワーク

### Step 1: Question（問いの定義）

何を知りたいのか。なぜ知りたいのか。

```markdown
- 分析目的: [何を明らかにするか]
- ビジネス文脈: [なぜこの分析が必要か]
- 意思決定への影響: [この分析結果がどの判断に使われるか]
- 期待する粒度: [日次/週次/月次、ユーザー単位/セグメント単位]
```

問いが曖昧なまま分析を始めない。曖昧な場合はCEO/依頼者に確認する。

### Step 2: Hypothesis（仮説の設定）

データを見る前に仮説を立てる。仮説なしの探索は「データの中をさまよう」だけになる。

```markdown
- 仮説1: [主仮説 -- 最も可能性が高いと考える説明]
- 仮説2: [対抗仮説 -- 主仮説が間違っている場合の説明]
- 棄却条件: [どのデータが出たら仮説を棄却するか]
```

### Step 3: Data（データ収集と品質評価）

データ品質アセスメントを実施してからAnalysisに進む。

#### データ品質チェックリスト

| 次元 | チェック内容 | 判定基準 |
|------|------------|---------|
| **完全性** | 欠損値の割合 | 主要フィールドの欠損率 < 5% |
| **正確性** | 一次ソースとの突合 | サンプル10件以上で一致 |
| **適時性** | データの鮮度 | 分析期間をカバーしているか |
| **一貫性** | 定義の統一 | 同一指標が同一定義で計算されているか |
| **網羅性** | 対象の網羅 | テストユーザー除外、全セグメントカバー |

#### データソース優先順位

```
1. Luna DB直接（一次ソース・最も信頼できる）
2. LROS metrics_monthly（SSoT・定義が明確）
3. Redash保存クエリ（検証済みクエリ）
4. アドホッククエリ（要検証）
```

LROS/NOVA/ARISの加工後データは正しさの根拠にならない。必ず一次ソースと突合する。

### Step 4: Analysis（分析の実行）

仮説に対してデータで検証する。

```markdown
- 使用データ: [データソース、クエリID、期間]
- 分析手法: [集計/比較/回帰/コホート等]
- 検証結果: [仮説は支持されたか/棄却されたか]
```

### Step 5: Conclusion（結論の導出）

```markdown
### 事実（データから直接読み取れること）
- [事実1]
- [事実2]

### 示唆（事実から導かれるビジネス示唆）
- [示唆1]
- [示唆2]

### 限界（このデータだけでは判断できないこと）
- [限界1]
- [限界2]

### 推奨アクション
- [アクション1]
```

## 2. 誤読防止チェック（7項目・必須）

全分析レポートに以下のチェック結果を記載する。省略不可。

### 2.1 期間ズレ

- **典型的な誤読**: 28日 vs 31日の比較で「減少」と誤判断
- **チェック**: 比較期間の日数が一致しているか。一致しない場合は日平均で比較
- **例**: 2月（28日）vs 3月（31日）の売上比較 → 日平均で比較する

### 2.2 重複カウント

- **典型的な誤読**: JOINによる行膨張で「増加」と誤判断
- **チェック**: JOIN前後のレコード数を比較。1:N結合で膨張していないか
- **例**: ユーザーテーブル x 注文テーブルのJOINで、複数注文ユーザーが重複カウント

### 2.3 欠損データ

- **典型的な誤読**: ログ欠損期間を「ゼロ」と誤判断
- **チェック**: 時系列の連続性を確認。不自然なゼロ値がないか
- **例**: 12/25-12/27のアクセスログが欠損 → 「年末にアクセスが激減」は誤り

### 2.4 テストユーザー

- **典型的な誤読**: テストデータ混入で指標が歪む
- **チェック**: テストユーザー・内部アカウントを除外しているか
- **例**: 社内テストアカウントのヘビーユースがDAU/MAUを過大評価

### 2.5 税抜/税込混在

- **典型的な誤読**: 売上の過大/過小評価
- **チェック**: 金額データの税込/税抜が統一されているか
- **例**: App Store経由（税込）とWeb経由（税抜）の売上を単純合算

### 2.6 サバイバーシップバイアス

- **典型的な誤読**: 離脱ユーザーを除外して「改善」と誤判断
- **チェック**: 分析対象に離脱/退会ユーザーが含まれているか
- **例**: 「アクティブユーザーの満足度が向上」→ 不満なユーザーは既に離脱済み

### 2.7 シンプソンのパラドックス

- **典型的な誤読**: 全体と部分で傾向が逆転していることに気づかない
- **チェック**: 全体の傾向がセグメント別でも一致するか
- **例**: 全体CVRは改善だが、男女別に見ると両方悪化（構成比の変化が原因）

## 3. 統計的妥当性の基礎

### サンプルサイズの判断

| N | 信頼性 | 扱い |
|---|--------|------|
| N < 30 | 低い | 「参考値」として記載。判断の根拠にしない |
| 30 <= N < 100 | 中程度 | 信頼区間を明示した上で使用 |
| N >= 100 | 高い | 通常の分析に使用可 |

### 信頼区間の明示

N < 100 の分析では必ず95%信頼区間を記載する。

```
例: CVR = 15.2% (95%CI: 12.1% - 18.3%, N=85)
→ 「CVRは12.1%から18.3%の範囲にある可能性が高い」
```

### 統計的有意性

- p < 0.05 を有意とする（一般的基準）
- p値だけでなく効果量（effect size）も報告する
- 「統計的に有意」と「実務的に意味がある」は別物

## 4. 可視化の選択ガイド

| 目的 | チャートタイプ | 適用場面 |
|------|-------------|---------|
| **時系列の推移** | 折れ線グラフ | MAU推移、売上推移、KPI推移 |
| **カテゴリ比較** | 棒グラフ | プラン別売上、セグメント比較 |
| **構成比** | 積み上げ棒/面グラフ | プラン構成比の推移、売上構成 |
| **相関関係** | 散布図 | 機能利用頻度 vs 継続率 |
| **分布** | ヒストグラム/箱ひげ図 | 利用時間の分布、課金額の分布 |
| **多次元比較** | ヒートマップ | セグメント x 指標のマトリクス |
| **フロー** | ファネルチャート | 登録→課金→継続の転換率 |

### 可視化の原則

- 1つのチャートに1つのメッセージ
- Y軸のゼロ起点を維持（例外を明示する場合のみ変更可）
- 色は意味を持たせる（赤=悪化、緑=改善、灰=基準）

## 5. 分析の落とし穴

### 5.1 サバイバーシップバイアス

生存者のみを分析対象にすることで、脱落者の情報を失う。

```
誤: 「3ヶ月継続ユーザーの満足度は90%」
正: 「3ヶ月以内に60%が離脱。残った40%の満足度は90%」
```

### 5.2 シンプソンのパラドックス

全体の傾向とサブグループの傾向が逆転する現象。

```
誤: 「全体のCVRが5%→6%に改善」
正: 「男性CVR 8%→7%、女性CVR 3%→2.5%、どちらも悪化。
     女性比率の増加（低CVR層の拡大）が全体CVRを押し上げた」
```

### 5.3 相関と因果の混同

2つの指標が連動しても、一方が他方の原因とは限らない。

```
誤: 「プロフィール充実度が高いユーザーはマッチ率が高い。プロフィール充実を促進すべき」
正: 「プロフィール充実度とマッチ率は相関するが、因果は未検証。
     モチベーションの高いユーザーが両方を行っている可能性がある」
```

### 5.4 チェリーピッキング

自説に都合のよい期間やデータだけを選択する。

```
誤: 「直近1週間でDAUが20%増加。施策が効いている」
正: 「直近1週間でDAUが20%増加。ただし前月の低迷期との比較であり、
     3ヶ月前と比較すると横ばい」
```
