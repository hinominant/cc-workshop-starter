# Prompt Caching Protocol

長時間エージェントセッションのコスト最適化戦略。
Anthropic API の prompt caching 機能を活用し、繰り返し送信されるコンテキストのコストを削減する。

> **Source**: Thariq (@trq212) "Prompt Caching Is Everything" — Claude Code 開発者の設計思想

---

## なぜ重要か

| 項目 | キャッシュなし | キャッシュあり |
|------|-------------|-------------|
| システムプロンプト（毎ターン） | 毎回フル課金 | 初回のみ。以降 90% 割引 |
| CLAUDE.md + _framework.md | 毎回フル送信 | キャッシュヒットで 0.1x コスト |
| 長いファイル読み込み | 読むたびフル課金 | 2回目以降はキャッシュ |
| 10ターンのセッション | $X | **約 $X × 0.3**（70% 削減） |

---

## キャッシュが効くもの・効かないもの

### キャッシュが効く（固定的なコンテキスト）

| コンテキスト | 特徴 | 戦略 |
|-------------|------|------|
| システムプロンプト | 全ターン共通 | **最優先でキャッシュ** |
| CLAUDE.md / _framework.md | セッション中は不変 | 先頭に配置してキャッシュ |
| .agents/PROJECT.md | セッション中はほぼ不変 | 先頭に配置 |
| エージェント SKILL.md | セッション中は不変 | キャッシュ対象 |
| 大きなコードファイル | 編集しなければ不変 | Read 後キャッシュ |

### キャッシュが効かない（変動するコンテキスト）

| コンテキスト | 特徴 | 戦略 |
|-------------|------|------|
| ユーザーメッセージ | 毎回異なる | キャッシュ不可 |
| ツール実行結果 | 毎回異なる | キャッシュ不可 |
| git diff 出力 | 毎回変わる | 最小限に絞る |
| エラーメッセージ | 毎回異なる | 必要分のみ |

## 設計原則

### 1. 固定コンテキストを先頭に配置

```
[キャッシュされる部分（先頭）]
├── システムプロンプト
├── CLAUDE.md
├── _framework.md
├── SKILL.md（使用中のエージェント）
├── PROJECT.md
└── DESIGN.md（存在する場合）

[キャッシュされない部分（後半）]
├── ユーザーメッセージ
├── ツール実行結果
└── アシスタント応答
```

### 2. コンテキストの順序を変えない

キャッシュはプレフィックス一致で機能する。途中にテキストを挿入すると、その位置以降の全キャッシュが無効になる。

**やってはいけないこと:**
- セッション途中で CLAUDE.md を書き換える（キャッシュ全破棄）
- エージェント SKILL.md をターンごとに入れ替える

**やるべきこと:**
- セッション開始時にコンテキストを固定し、以降は変更しない
- 追加情報はメッセージの末尾に追加する

### 3. 大きなファイルは1回だけ Read

同じファイルを複数回 Read するとキャッシュが効くが、**最初の Read で必要な情報を全部取る**方がトークン効率が良い。

```
❌ Bad: Read file → 作業 → Read same file → 作業 → Read same file
✅ Good: Read file（必要部分メモ）→ 作業 → 作業 → 作業
```

### 4. エージェントチェーンのキャッシュ戦略

Nexus → Builder → Radar のチェーンで、各エージェントに渡すコンテキストを最小化:

| チェーン段階 | 渡すもの | 渡さないもの |
|-------------|---------|-------------|
| Nexus → Builder | タスク要件 + 関連ファイルパス | Nexus の内部分析ログ |
| Builder → Radar | 変更ファイルリスト + テスト方針 | Builder の実装試行錯誤 |

**SLIM_CONTEXT.md と併用**: トークン予算管理で不要なコンテキストを除去。

## コスト見積もり

### Sonnet（標準エージェント）

| ターン数 | キャッシュなし | キャッシュあり | 削減率 |
|---------|-------------|-------------|-------|
| 5 | ~15,000 入力トークン×5 = 75k | 15k + 4k×4 = 31k 実効 | 59% |
| 10 | ~150k | ~55k 実効 | 63% |
| 20 | ~300k | ~95k 実効 | 68% |

### Opus（CEO/設計判断）

Opus は単価が高いため、キャッシュの効果がさらに大きい。
CEO セッションは短く保つ（5ターン以内推奨）。

---

## エージェントへの適用ルール

| ルール | 説明 |
|--------|------|
| **セッション開始時にコンテキストを固定** | CLAUDE.md + SKILL.md + PROJECT.md を最初に読み、以降変更しない |
| **Read は1回で完結** | 同じファイルの再読み込みを避ける |
| **チェーン間は SLIM_CONTEXT** | 前エージェントの全出力を渡さない。要約のみ |
| **長時間セッションは compaction 前に判断** | 50% 到達で manual compact 推奨（CONTEXT_HYGIENE.md） |
| **Opus は短時間で決断** | CEO セッションは 5ターン以内。長引くなら Sonnet にエスカレート |
