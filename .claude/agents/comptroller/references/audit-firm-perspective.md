## 1. 監査法人がIT統制を評価する方法

### 1.1 評価の全体フレームワーク

```
[監査法人のIT統制評価プロセス]

Phase 1: 計画（Planning）
  ├── IT環境の理解（システム構成、データフロー）
  ├── 重要なアプリケーションの特定（財務報告に係るもの）
  └── リスク評価とスコーピング

Phase 2: 設計評価（Design Effectiveness）
  ├── 統制が設計上リスクに対応しているか
  ├── 統制の実行者・頻度・方法が適切か
  └── RCM（リスクコントロールマトリクス）の網羅性

Phase 3: 運用評価（Operating Effectiveness）
  ├── 統制が期間を通じて一貫して運用されているか
  ├── サンプリングテスト（25件以上が標準）
  └── 証跡による実証手続

Phase 4: 不備評価（Deficiency Evaluation）
  ├── 識別された不備の重要度分類
  ├── 財務報告への影響度分析
  └── 「重要な不備」の該当性判断
```

### 1.2 監査法人が重視する3つの「距離」

| 距離 | 内容 | comptrollerの検証方法 |
|------|------|---------------------|
| 規程と運用の距離 | 規程は整備されているが、現場で守られているか | 12ヶ月分の証跡をサンプリングし、規程通りに運用されているか確認 |
| 設計と実態の距離 | システム構成図・フロー図が実態と一致しているか | 実際のシステム設定と文書の突合 |
| 認識とリスクの距離 | 経営層のリスク認識と実際のリスクが一致しているか | リスク評価書の内容と直近のインシデント・脆弱性情報の照合 |

## 2. 監査法人の頻出指摘事項と期待水準

### 2.1 アクセス管理領域

| 指摘パターン | 監査法人の期待水準 | 典型的な不備 |
|-------------|------------------|-------------|
| 退職者ID管理 | 退職日当日の無効化。遅延は3営業日以内（許容範囲外は不備） | 退職後1ヶ月以上IDが有効、人事システムとIAMの連携不足 |
| 権限棚卸し | 四半期以上の頻度で全ユーザー対象。不要権限の是正記録を含む | 年1回のみ、棚卸し後の是正が未記録 |
| 特権アカウント | 利用は最小限、全利用のログ取得、定期レビュー | 特権アカウントの常時利用、ログ未取得 |
| SoD（職務分掌） | 開発/運用/承認の明確な分離。SoDマトリクスの文書化 | 開発者が本番環境にデプロイ可能、承認者と実施者が同一 |

### 2.2 変更管理領域

| 指摘パターン | 監査法人の期待水準 | 典型的な不備 |
|-------------|------------------|-------------|
| デプロイ承認 | 全本番デプロイに事前承認。承認者は開発者以外 | 自動デプロイに承認ゲートなし、緊急対応の事後承認漏れ |
| テスト証跡 | テスト計画→実行→結果→承認の一貫した証跡チェーン | テスト結果のみでテスト計画なし、結果に承認がない |
| 環境分離 | 開発/ステージング/本番の物理的（又はネットワーク的）分離 | 開発環境から本番DBへの接続が可能 |
| 変更の追跡可能性 | 変更要求→承認→実装→テスト→デプロイの全トレース | PRとデプロイの紐付けが不完全 |

### 2.3 コンピュータ運用領域

| 指摘パターン | 監査法人の期待水準 | 典型的な不備 |
|-------------|------------------|-------------|
| バックアップ | 日次バックアップ + 四半期リストアテスト（成功記録必須） | バックアップは取得しているがリストアテスト未実施 |
| 監視 | 重要システムの24/365監視、異常検知時の自動アラート | 営業時間のみの監視、アラートの閾値が未設定 |
| インシデント管理 | 発生→対応→原因分析→再発防止の一貫したプロセス | 対応記録はあるが原因分析・再発防止が未実施 |
| ジョブ管理 | 定期ジョブの実行監視、失敗時のアラートと対応手順 | ジョブ失敗の検知が遅延、対応手順が未文書化 |

## 3. 証跡の品質基準

### 3.1 監査法人が求める証跡の4要件

| 要件 | 説明 | NG例 |
|------|------|------|
| **完全性** | 対象期間の全件が網羅されていること | 一部の月だけ、特定イベントのみ |
| **正確性** | データが改ざんされていない、タイムスタンプが正確 | 事後作成の証跡、日時が不自然 |
| **適時性** | 統制が実行された時点に近い証跡であること | 事後にまとめて作成された証跡 |
| **独立性** | 自己申告ではなくシステムから自動取得された証跡 | 手書きのチェックリスト、口頭報告 |

### 3.2 証跡の種類別品質ランク

| ランク | 証跡の種類 | 監査法人の評価 |
|--------|-----------|--------------|
| A（最高） | システム自動生成ログ（CloudTrail、CI/CDログ、監査ログ） | 改ざん困難、タイムスタンプ信頼性高 |
| B（高） | システムからのエクスポート（CSV、JSON） + ハッシュ値 | エクスポート時点の整合性を確認可能 |
| C（中） | スクリーンショット + タイムスタンプ | 部分的な証跡、前後関係の確認が困難 |
| D（低） | 手書き記録、自己申告チェックリスト | 監査法人はこれだけでは不十分と判断する可能性 |

### 3.3 スクリーンショット証跡の要件

スクリーンショットを証跡として使用する場合の最低要件:

- [ ] 日時が画面内に表示されている（OSの時計 or システムのタイムスタンプ）
- [ ] 対象システムの特定が可能（URL、システム名が表示）
- [ ] 操作内容が明確（設定画面であれば設定値が読み取れる）
- [ ] 前後のコンテキストが理解可能（何を確認したかの説明文を付記）
- [ ] 連番管理されている（証跡#001、#002...）

### 3.4 ログ証跡の要件

| 要件 | 基準 |
|------|------|
| 保存期間 | 最低5年（J-SOX対象期間 + 監査対応期間） |
| 改ざん防止 | Write-once storage、またはハッシュチェーン |
| アクセス制御 | ログへのアクセス権限が制限されている（セキュリティ/コンプライアンス担当のみ） |
| 完全性 | ログの欠落がないこと（ログローテーション時のギャップ防止） |
| 検索性 | 監査時に必要な情報を迅速に検索・抽出できること |

## 4. 重要性（Materiality）の判断基準

### 4.1 IT統制不備の重要度分類

| 分類 | 基準 | 影響 |
|------|------|------|
| **重要な不備（Material Weakness）** | 財務報告の重要な虚偽表示が防止・発見されない合理的な可能性がある | 内部統制報告書に「開示すべき重要な不備」として記載。上場審査では致命的 |
| **不備（Deficiency）** | 統制が設計・運用上の目的を達成しない。単独では重要な虚偽表示のリスクは低い | 改善勧告。複数の不備の組み合わせで重要な不備になりうる |
| **軽微な不備（Minor Deficiency）** | 統制の有効性に影響しない程度の不備 | 内部改善として対応 |

### 4.2 不備の重要度判定フロー

```
[不備の重要度判定]

1. 財務報告への影響度を評価
   ├── 影響する勘定科目は何か
   ├── 影響金額はいくらか（推定）
   └── 影響の発生可能性はどの程度か

2. 代替統制の存在を確認
   ├── 不備をカバーする代替統制があるか
   └── 代替統制は有効に機能しているか

3. 補完的手続きの可能性
   ├── 追加的なモニタリングで不備を補完可能か
   └── 補完的手続きの証跡があるか

4. 総合判定
   ├── 影響大 + 代替なし → 重要な不備
   ├── 影響中 + 代替あり → 不備
   └── 影響小 or 代替十分 → 軽微な不備
```

## 5. サンプリング方法論

### 5.1 IT監査のサンプリング基準

| 統制の頻度 | 母集団サイズ | 最低サンプル数 |
|-----------|------------|--------------|
| 日次（年間250件以上） | 250+ | 25件 |
| 週次（年間52件） | 52 | 15件 |
| 月次（年間12件） | 12 | 5件（又は全件） |
| 四半期（年間4件） | 4 | 全件 |
| 年次（1件） | 1 | 全件 |
| 都度（イベントドリブン） | 変動 | 25件（母集団250以上）/ 全件（250未満は判断） |

### 5.2 サンプル抽出方法

| 方法 | 説明 | 適用場面 |
|------|------|---------|
| ランダムサンプリング | 母集団から無作為に抽出 | 均質な母集団（定期的な統制） |
| 層別サンプリング | 母集団を層に分けて各層から抽出 | 金額や重要度にばらつきがある場合 |
| 意図的サンプリング | 特定の条件（リスクが高い期間、大規模変更）に該当する項目を選択 | 特定のリスクシナリオの検証 |
| 全件検証 | 母集団の全件を検証 | 年間件数が少ない統制、CRITリスクの統制 |

### 5.3 サンプリングテストで不備が見つかった場合

```
不備の検出
│
├── 1件の不備 → 追加サンプル（最低10件）を抽出し再テスト
│   ├── 追加テストで不備なし → 例外的事象として記録
│   └── 追加テストで不備あり → 統制の運用有効性に疑義
│
├── 2件以上の不備 → 統制の運用有効性なしと判断
│   ├── 代替統制の有無を確認
│   └── 不備の影響度分析を実施
│
└── comptrollerの判断
    ├── complianceが「問題なし」としていないか確認
    ├── サンプリング方法が適切だったか検証
    └── 母集団の定義に漏れがないか確認
```

## 6. comptroller固有の検証チェックリスト

complianceの評価結果をレビューする際、以下を必ず確認する:

| # | 確認事項 | 確認方法 |
|---|---------|---------|
| 1 | チェックリストの全項目が評価されているか | 項目番号の突合 |
| 2 | 「問題なし」の項目に証跡が添付されているか | 証跡の存在確認 |
| 3 | 証跡のタイムスタンプが評価対象期間内か | 日付の確認 |
| 4 | 重要度判定がIPO審査/J-SOX基準と整合しているか | 本リファレンスの判定基準と照合 |
| 5 | 是正勧告が具体的かつ実現可能か | 是正内容の実現可能性評価 |
| 6 | AI利用に関する評価が最新のガイドラインに基づいているか | AI事業者ガイドライン版数の確認 |
| 7 | 委託先IT統制（SOCレポート等）の評価が含まれているか | J-SOX 2024年改訂要件の充足確認 |
