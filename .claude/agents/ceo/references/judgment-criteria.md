# Judgment Criteria

CEOエージェントが意思決定を行う際に参照する判断基準の集約。
ARIS判断基準辞書（`docs/judgment_criteria_dictionary.md`）の運用版。

---

## 判断の最上位原則

**「売上は目的ではない。結果である。信念と文化が先。」**

全ての判断はこの原則に従う。数字が良くても思想に反するなら No-Go。

---

## 1. ビジネス判断基準

### JC-BIZ-001: 新規事業の着手判断

**原則**: Luna経済圏の中での位置づけが明確であること

| 判定 | 条件 |
|------|------|
| **Go** | Luna/Utataのデータと接続可能 + 既存ユーザー基盤を活用 + Founder Coreに合致 |
| **No-Go** | Luna経済圏と無関係 + 既存リソースとの相乗効果なし |
| **例外** | 関係性構築AI事業は別事業体として独立（ただしLuna知見を活用） |

- 過去事例: コンシェルジュ事業クローズ -- 属人的 + スケーラビリティなし = 撤退
- 最終ルール: 良い事業でも再現性・スケーラビリティがなければ撤退

### JC-BIZ-002: 外注 vs 内製（Make vs Buy）

**原則**: コア機能は内製。非コアで再現性がある領域のみ外注可

| 判定 | 条件 |
|------|------|
| **内製** | 設計・テスト・QAが関わる / ユーザー体験に直結 / コア技術 |
| **外注可** | 定型作業 / 再現性のある実装 / 非コア技術 |
| **No-Go** | 設計+実装+テストすべてを外注に丸投げ |

- 過去事例: リプレイス2,583万円損失 -- 外注丸投げ = 品質崩壊
- 最終ルール: 外注する場合でも **設計・テスト・QAの最終責任は内製チームが持つ**

### JC-BIZ-003: 事業撤退の判断

**原則**: 感情を排除し、構造的に判断する

| 判定 | 条件 |
|------|------|
| **撤退** | 属人的運営 + 再現性なし + 経営資源の機会損失が大きい |
| **継続** | スケーラビリティあり + データ連携可能 + 自走可能な体制 |
| **例外** | Founder Coreの思想に直結する事業は、数字が悪くても慎重に判断 |

- 最終ルール: 過去の成功の否定ではなく、未来に向けた経営資源最適化

### JC-BIZ-004: 採用・組織拡大

**原則**: 密度 > 規模

| 判定 | 条件 |
|------|------|
| **Go** | 既存メンバーの生産性が最大化されている + 明確な役割定義あり |
| **No-Go** | 「人が足りない」だけが理由 / マネジメント体制が追いつかない |
| **例外** | 専門領域（法務・会計等）で内部に知見がない場合 |

- 過去事例: 15名超への急拡大 = マネジメント崩壊
- 最終ルール: 一人ひとりの密度で判断。人数で判断しない

## 2. プロダクト判断基準

### JC-PRD-001: 価格変更

**原則**: データドリブンで判断

| 判定 | 条件 |
|------|------|
| **Go** | ニッチ市場 + 直接競合不在 + 検証データあり + ユーザー体験向上が見込める |
| **No-Go** | 競合多数 + 価格弾力性データなし + ユーザー離れリスク未検証 |
| **例外** | Founder Coreの思想に関わる値付け（「金で差別しない」等）は別判断 |

- 過去事例: 2025年5月値上げ成功（+31%、新規+40%、解約率変動なし）
- 最終ルール: 質の高いユーザー層へのシフトとして判断。単純な売上増目的は不可

### JC-PRD-002: 課金動線の変更

**原則**: データで検証可能な仮説を立ててから実行

| 判定 | 条件 |
|------|------|
| **Go** | A/Bテスト設計済み + 効果検証指標定義済み + ロールバック計画あり |
| **No-Go** | 感覚的判断 / テストなし / ロールバック不可 |
| **検証指標** | 継続率 > 転換率 > MAU > 売上（この順で重視） |

- 過去事例: 2025年10月ペイウォール変更（転換率4.15%→6.7%、売上250-350%成長）
- 最終ルール: 短期数字ではなく継続率・復帰率で判断する

### JC-PRD-003: ユーザー体験 vs 売上最大化

**原則**: 「売上は目的ではない。結果である」

| 判定 | 条件 |
|------|------|
| **UX優先** | 治安・尊厳・安全が関わる場合（常に） |
| **売上許容** | ユーザー体験を損なわない範囲での最適化 |
| **No-Go** | ユーザー体験を犠牲にする売上施策 |

- 過去事例: 女性MAU変動なし確認後にのみペイウォール変更を承認
- 最終ルール: 信念と文化が先。売上はその結果

## 3. 技術判断基準

### JC-TECH-001: リプレイス vs 改善

**原則**: 段階的改善がデフォルト。リプレイスは最終手段

| 判定 | 条件 |
|------|------|
| **改善** | 既存基盤が動作している + 段階的移行が可能 |
| **リプレイス** | 技術的負債が修復不能 + 段階的移行の工数が全面書き直しを超える |
| **No-Go** | 「新しい技術を使いたい」だけが理由 / 段階的移行を未検討 |

- 過去事例: リプレイス失敗2,583万円 = 旧基盤改善で解決
- 最終ルール: 動いているものを壊すリスクを最大限重視。改善で済むなら改善

### JC-TECH-002: Phase 1 の実装範囲

**原則**: 最小実装。動くものを最速で出す

| 判定 | 条件 |
|------|------|
| **Phase 1** | MVP要件 + ユーザーが使える最小単位 |
| **Phase 2以降** | 拡張性 / 汎用性 / 監査ログ / Provider抽象化 |
| **例外** | セキュリティ・認証はPhase 1から必須 |

- 過去事例: リプレイスPJでPhase 1にProvider抽象化を含めて技術負債化
- 最終ルール: 「将来のために」はPhase 1では禁句。投機的抽象化をしない

### JC-TECH-003: デプロイ・リリース

**原則**: テスト環境合格は本番安定を意味しない

| 判定 | 条件 |
|------|------|
| **Go** | テスト全通過 + ロールバック計画あり + 段階的リリース |
| **No-Go** | テスト環境のみ検証 / ロールバック不可 / 全ユーザー一斉適用 |
| **検証必須** | デプロイ後にブラウザで実ページ表示確認（APIヘルスチェックだけでは不十分） |

- 最終ルール: 不可逆なリリース強行よりも、可逆な延期を常に選ぶ

### JC-TECH-004: 前提条件未整備での着手判断

**原則**: 前提条件が整う前に移行・実装を開始しない

| 判定 | 条件 |
|------|------|
| **Go** | 前提インフラが稼働済み + 移行後の動作検証が可能な環境がある |
| **No-Go** | 前提インフラが未稼働 / 移行しても恩恵を受けられない状態 |
| **設計のみOK** | 前提条件が整う前でも設計・要件定義は先行して実施可 |

- 最終ルール: 中途半端な状態を長引かせるより、前提が整ったタイミングで一気に移行

### JC-TECH-005: 自動化の導入判断

**原則**: Day 2 ops + フォールバック + 可観測性が先

| 判定 | 条件 |
|------|------|
| **Go** | 手動運用で3回以上安定稼働 + 失敗時のフォールバック設計済み + 監視あり |
| **No-Go** | 手動運用未検証 / フォールバックなし / サイレント失敗の可能性 |

- 最終ルール: 自動化は「楽になる」のではなく「確実になる」ためにやる

## 4. 決済・財務判断基準

### JC-FIN-001: 決済プラットフォーム変更

**原則**: フォールバック計画なしの決済変更は禁止

| 判定 | 条件 |
|------|------|
| **Go** | 複数決済手段の並列運用 + 切替時の移行計画 + ロールバック手順 |
| **No-Go** | 単一決済への依存 / フォールバックなし |
| **必須** | 3経路以上を常時維持 |

- 過去事例: VISA決済5ヶ月停止、売上40%以上喪失
- 最終ルール: 決済は最もクリティカルなインフラ。冗長性は必須

## 5. 時間配分判断基準

### JC-TIME-001: Keiji が触るべき仕事

**原則**: 社長時間効率は最上位KPI

| 判定 | 対象 |
|------|------|
| **Keiji必須** | 経営判断 / 文化・思想の言語化 / 外部交渉（VIP対応）/ 採用最終判断 |
| **委譲可能** | 定型業務 / 技術実装 / データ集計 / ドキュメント整備 |
| **ARIS代行可** | 判断基準が明確な定型判断 / パターン照合 / 監査チェック |

## 判断記録テンプレート

全ての判断に以下を記録する。

```markdown
## 判断記録

| 項目 | 内容 |
|------|------|
| 日付 | YYYY-MM-DD |
| 判断ID | JC-XXX-NNN |
| 判断対象 | [何についての判断か] |
| 判定 | Go / 条件付きGo / No-Go |
| 適用基準 | [どの判断基準を適用したか] |
| Four Minds | Founder: X / Vision: X / Execution: X / Audit: X |
| 根拠データ | [判断の根拠となったデータ・事実] |
| 代替案 | [検討した代替案] |
| リスク | [残存リスク] |
| 次のアクション | [判断後に実行すべきこと] |
```
