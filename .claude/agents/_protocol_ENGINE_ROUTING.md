# Engine Routing Protocol

タスクの性質に応じて Claude Code / Codex / Gemini を最適に振り分けるルール。
**得意分野ファーストの設計** — クレジット保全はその副産物として得る。

---

## 設計思想

```
目的①: タスクを最も得意なエンジンに振り分け、品質と速度を最大化
目的②: Claude Code を高判断タスクに集中させ、weekly limit を自然に温存
フォールバック: weekly limit 到達時のみ、下位エンジンに引き継ぐ
```

---

## エンジン特性サマリー

| エンジン | 強み | 弱み |
|---------|------|------|
| **Claude Code** | 複雑な推論・文脈理解・設計判断・ビジネスロジック | クレジット消費が大きい、weekly limit あり |
| **Codex** (o4-mini / o3) | アルゴリズム・コード生成・パターン適用・反復タスク | 仕様の曖昧さに弱い・ビジネス文脈を持たない |
| **Gemini** | 広大なコンテキスト処理・文章生成・探索的分析 | 細かなコード品質・セキュリティ判断は弱め |

---

## ルーティング表

### Claude Code PRIMARY

> 判断・文脈・設計の質がプロダクトに直結するタスク。エスカレーション不要。

| タスク | 理由 |
|--------|------|
| アーキテクチャ設計・システム設計 | システム全体の整合性判断が必要 |
| 探索的設計（仕様が曖昧） | 曖昧さを推論で埋める能力が必要 |
| コードレビュー・最終品質評価 | 文脈理解と多角的判断が必要 |
| セキュリティ・認証・課金処理の実装 | ミスの影響が大きく品質妥協不可 |
| 複雑なデバッグ（原因不明・多系統） | 広い推論と仮説立案が必要 |
| ビジネスロジック実装（Luna固有の文脈） | ドメイン知識をコードに反映する必要 |
| 複数ファイルにまたがる依存が複雑な変更 | 整合性の維持に深い理解が必要 |
| 他エンジンの出力レビュー・統合 | 最終品質ゲートは Claude Code が担う |

---

### Codex PRIMARY

> 仕様が固まっていて、コードに集中できるタスク。
> **条件**: 仕様が完全に確定していること。曖昧さが発見されたら即エスカレーション。

| タスク | エスカレーション条件（Claude Code へ上げる） |
|--------|---------------------------------------------|
| アルゴリズム・データ構造の実装 | 仕様に未定義の分岐・エッジケースが発見された場合 |
| テストコード生成 | テスト戦略の設計判断が必要になった場合 |
| 機械的リファクタリング（パターン明確） | リファクタ中に設計判断が発生した場合 |
| 機能実装（仕様が100%確定している） | ①仕様の曖昧さを発見 ②変更ファイルが3件超かつ依存が複雑 ③セキュリティ・認証・課金に触れる |
| バグ修正（再現手順が明確・テストあり） | ①3ループ以上修正できない → o3 に切替 → さらに2ループ直らない → Claude Code ②根本原因がアーキテクチャ起因と判明した場合 |
| ボイラープレート・スキャフォールド生成 | カスタムロジックの判断が必要になった場合 |

**Codex モデル選択指針:**

| タスク | 推奨モデル |
|--------|-----------|
| バグ修正ループ・繰り返し作業・lint | `o4-mini`（デフォルト・高速） |
| 複雑なアルゴリズム・精度が必要 | `o3` |

**バグ修正ループパターン（o4-mini 推奨）:**

```bash
# テストが通るまで全自動で高速反復する
codex exec --full-auto -m o4-mini \
  "テストを実行し、失敗しているテストを修正して再実行。全テストが通るまで繰り返す。"
```

**反復エスカレーションルール:**

```
o4-mini で実行
  └─ 3ループ以内に解決 → 完了
  └─ 3ループ超えても未解決 → o3 に切替
      └─ 2ループ以内に解決 → 完了
      └─ 2ループ超えても未解決 → Claude Code にエスカレーション
          └─ 根本原因分析から再実施
```

---

### Gemini PRIMARY

> 広いコンテキストと文章生成が価値を出すタスク。

| タスク | エスカレーション条件（Claude Code へ上げる） |
|--------|---------------------------------------------|
| ドキュメント・README・仕様書の生成 | 正確性の検証にコードの深い理解が必要な場合 |
| 大規模コードベースの調査・把握 | 調査結果を元に実装・設計判断が必要になった場合 |
| 設計メモ・ミーティングノートの整形・要約 | 内容の正誤判断に専門知識が必要な場合 |
| 多ファイルにまたがる影響分析（読み取り専用） | 分析結果に基づく変更判断が必要な場合 |

## フォールバックチェーン（weekly limit 到達時のみ）

```
Claude Code ──limit──▶ Codex ──limit──▶ Gemini ──limit──▶ 通知: 全エンジン停止
```

**フォールバック発動条件:**
- Claude Code の weekly limit に到達した場合
- エンジン CLI が利用不可の場合（`codex` or `gemini` コマンド not found）

**フォールバック時の品質注意:**
- Codex に引き継いだ場合、仕様を明確化してから渡すこと
- Gemini に引き継いだ場合、コード変更タスクは慎重に（品質レビューを必須化）

## エスカレーション判定フロー

```
タスク受信
  │
  ▼
[1] タスクは Claude Code PRIMARY に該当するか？
  │ YES → Claude Code で実行
  │ NO ↓
  ▼
[2] 仕様は100%確定しているか？
  │ NO → Claude Code で仕様確定を先に行う
  │ YES ↓
  ▼
[3] セキュリティ・認証・課金に触れるか？
  │ YES → Claude Code
  │ NO ↓
  ▼
[4] 変更ファイルは3件以下かつ依存が単純か？
  │ NO → Claude Code
  │ YES ↓
  ▼
[5] タスク種別で振り分け
  ├── コード生成・アルゴリズム・テスト → Codex
  └── ドキュメント・調査・要約 → Gemini
```

## Nexus / Rally 実行テンプレート

```yaml
# タスク投入時の記載形式
ENGINE_ROUTING:
  task_description: "LROSのバックフィルスクリプトを実装する"
  bloom_level: L3 (APPLY)
  spec_status: confirmed          # confirmed / unclear
  touches_security: false
  file_change_count: 2
  primary_engine: codex
  model: o4-mini
  escalation_trigger: "仕様に曖昧さが発見された場合は Claude Code へ"
  agent: Arena → (Codex exec) → Radar
```

---

## プロジェクト固有の上書き

プロジェクト直下の `.agents/ENGINE_ROUTING.md` でデフォルトを上書き可能。
ない場合はこのグローバル設定が適用される。

上書き例（高複雑プロジェクト向け）:
```md
## ENGINE_ROUTING Override
- 機能実装（仕様明確）: Codex PRIMARY → Claude Code PRIMARY
- エスカレーション閾値: 全般的に1段低く（より積極的に Claude Code へ）
```

---

## エンジン別実行コマンド（Arena経由）

```bash
# Codex
codex exec --full-auto -m o4-mini "implement: {spec}"
codex exec --full-auto -m o3 "implement: {spec}"   # 精度重視時

# Gemini
gemini -p "{prompt}"

# Claude Code（通常のセッション内実行）
# → Builder / Scout 等の通常エージェントが担当
```
