## 1. Counselの法的分析レビュー方法

### 1.1 レビューの前提

counselの法的分析は以下の構造で提出される:
- 適用法令の特定と準拠状況
- 契約リスク評価
- AI法務リスク評価
- 知的財産リスク評価

advocateは各領域について独立した検証を行う。**counselの結論を出発点としない。同じ素材から独自に結論を導き、その上で比較する。**

### 1.2 法的分析の検証7ステップ

```
Step 1: 法令特定の網羅性確認
  → counselが適用法令を見落としていないか
  → 業種特有の規制（金融業法、薬機法、景品表示法等）の漏れがないか
  
Step 2: 条文引用の正確性検証
  → 条文番号が正しいか（e-Gov法令検索で原文突合）
  → 最新改正が反映されているか
  → 施行令・施行規則の参照漏れがないか
  
Step 3: 法的解釈の妥当性検証
  → 条文の射程（適用範囲）の解釈が適切か
  → 通説・判例と異なる独自解釈をしていないか
  → 例外規定・適用除外の見落としがないか
  
Step 4: リスク評価の較正
  → CRIT/HIGH/MED/LOWの判定が適切か
  → 罰則金額とリスクレベルの整合性
  → 事業への影響度が正しく反映されているか
  
Step 5: 代替解釈の検討
  → counselの解釈と異なる解釈の可能性
  → より保守的な解釈（リスク最小化）
  → より緩やかな解釈（事業機会確保）
  
Step 6: 是正勧告の実効性確認
  → 推奨された対策が法的に十分か
  → 過剰な対応を推奨していないか
  → 代替的な対応策はないか
  
Step 7: 最終判定
  → APPROVED: 全ステップクリア
  → RETURNED: 修正が必要な項目あり
```

## 2. 判例・先例検証テクニック

### 2.1 判例検索の方法論

| 検索戦略 | 方法 | 用途 |
|---------|------|------|
| 論点ベース検索 | 法的論点をキーワードに分解して検索 | counselの判断の法的根拠確認 |
| 条文ベース検索 | 引用条文を起点に関連判例を検索 | 条文解釈の確立度確認 |
| 類似事案検索 | 対象事業と類似の事業形態に関する判例 | リスク評価の較正 |
| 反対結論検索 | counselと逆の結論に至った判例を積極的に検索 | 反証の確認 |

### 2.2 判例の射程判断フレームワーク

```
判例の射程 = 以下4要素の総合判断

1. 事実関係の類似性
   ├── HIGH: 当事者の属性・行為・結果が類似
   ├── MED: 一部要素が類似
   └── LOW: 法的論点のみ共通

2. 法的論点の同一性
   ├── 同一条文の同一解釈問題か
   ├── 類似の利益衡量が必要か
   └── 保護法益が同じか

3. 裁判所のレベル
   ├── 最高裁判例 → 拘束力あり
   ├── 高裁判例 → 事実上の拘束力
   └── 地裁判例 → 参考情報

4. 時期的要素
   ├── 法改正前後で射程が変わる
   ├── 社会状況の変化（AI/DX等）
   └── ガイドライン制定前後
```

### 2.3 重要判例カテゴリ（IT企業関連）

| カテゴリ | 代表的判例・事案 | 検索キーワード |
|---------|----------------|---------------|
| 営業秘密侵害 | 転職者による顧客情報持ち出し | 不正競争防止法 営業秘密 IT |
| ソフトウェア著作権 | プログラムの著作物性、翻案権侵害 | 著作権 プログラム ソースコード |
| OSS契約違反 | GPL違反訴訟（海外判例含む） | GPL 著作権 オープンソース |

| 分析項目 | 確認内容 |

## 4. ギャップ分析テンプレート

### 4.1 Counselの結論 vs 法的現実のギャップ分析

```markdown
## GAP ANALYSIS REPORT

### 分析日: YYYY-MM-DD
### 対象: [counsel の評価対象]

### ギャップ一覧

| # | 領域 | Counsel結論 | Advocate検証結果 | ギャップ | 重要度 | 根拠 |
|---|------|-----------|----------------|---------|--------|------|
| 1 | [法令名] | [counselの判定] | [advocateの判定] | [差異の内容] | CRIT/HIGH/MED/LOW | [判例・処分事例番号] |

### ギャップ分類

| ギャップ類型 | 説明 | 対応方針 |
|-------------|------|---------|
| 法令解釈の相違 | 条文解釈が異なる | 判例・通説に基づき統一見解を形成 |
| リスク評価の相違 | 重要度判定が異なる | 罰則金額・行政処分事例に基づき再較正 |
| 法令適用の見落とし | counselが適用法令を見落とし | 追加評価を依頼 |
| 最新動向の未反映 | 法改正・ガイドライン更新の未反映 | 最新版での再評価を依頼 |
| 是正勧告の不十分 | 推奨対策が法的に不十分 | 追加対策を提案 |
```

### 4.2 ギャップの重要度判定基準

| 重要度 | 基準 | 対応 |
|--------|------|------|
| CRIT | 法令違反の可能性あり。罰則適用リスクあり。行政処分の先例あり | 即座にRETURNED。是正完了まで進行停止 |
| HIGH | ガイドライン非準拠。行政指導リスクあり。類似の処分事例あり | RETURNED。修正後に再検証 |
| MED | 法的リスクは低いが、ベストプラクティスから乖離 | 付帯意見として記載。APPROVEDとしうる |
| LOW | 推奨事項。法的義務ではないが改善が望ましい | 付帯意見として記載。APPROVED |

## 5. 意見相違の解決プロトコル

### 5.1 解決フロー

```
[意見相違発生]
│
├── Step 1: 相違点の明確化
│   └── counselの結論 vs advocateの結論を表形式で整理
│
├── Step 2: 根拠の比較
│   ├── 各結論の法的根拠（条文・判例・学説）を列挙
│   └── 根拠の強さを比較（最高裁判例 > 高裁 > 地裁 > 学説 > ガイドライン）
│
├── Step 3: リスクの比較
│   ├── 各結論を採用した場合のリスク（最悪シナリオ）を比較
│   └── 保守的立場（リスク最小化）の結論を特定
│
├── Step 4: 合意形成の試行
│   ├── 条件付き合意の可能性（「Xの場合はcounsel結論、Yの場合はadvocate結論」）
│   └── 追加調査で解決可能な論点の特定
│
└── Step 5: エスカレーション判断
    ├── 合意成立 → 合意内容を最終レポートに記載
    ├── 部分合意 → 合意点と未合意点を明記してNexusに報告
    └── 合意不成立 → 両論併記してユーザーに判断を求める
```

### 5.2 エスカレーション基準

| 条件 | 対応 |
|------|------|
| CRIT判定に相違がある | 必ずユーザーにエスカレーション |
| 罰則適用の判断が分かれる | 弁護士相談を推奨 |
| 判例がなく学説のみで判断が分かれる | 両論併記 + リスクシナリオを提示 |
| AI法務で先例がない論点 | 保守的立場を推奨 + 定期見直しを提案 |

### 5.3 最終レポートへの記載ルール

- 合意した結論: 統一見解として記載
- 相違が残る論点: 「主査見解」「副査見解」を明記し、それぞれの根拠を付す
- 推奨判断: 保守的立場（リスク最小化）を推奨として明記
- 免責: 「本評価は法的助言ではなく、最終判断は弁護士への相談を推奨します」を必ず付記
