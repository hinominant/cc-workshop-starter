# Error Recovery Pattern

## Overview

| 項目 | 値 |
|------|-----|
| Pattern | Error Recovery |
| Category | Feedback / Resilience |
| Complexity | Medium |
| Reference | SmartHR「エラーハンドリング」相当 |

**解決する課題:** ユーザー操作・通信・システムエラーが発生した際に、原因を明確に伝え、回復手順を案内して操作継続を支援する。

**使用場面:**
- APIリクエスト失敗（通信エラー・タイムアウト）
- フォームバリデーションエラー
- 認証エラー（セッション切れ等）
- ページ全体の読み込み失敗（Error Boundary）

---

## Structure

### Level 1: インラインエラー（フォームフィールド）

```
┌──────────────────────────────┐
│ メールアドレス                │
│ ┌────────────────────────┐  │
│ │ invalid@               │  │
│ └────────────────────────┘  │
│ ⚠ 正しいメールアドレスの形式で │
│   入力してください             │
└──────────────────────────────┘
```

### Level 2: トーストエラー（API失敗）

```
┌──────────────────────────────────┐
│ ✕  保存に失敗しました。もう一度    │
│    お試しください。               │
└──────────────────────────────────┘
（画面上部、2.5秒表示）
```

### Level 3: インラインバナーエラー（ページ内エラー）

```
┌──────────────────────────────────────┐
│ ⚠ データの取得に失敗しました           │
│   通信環境を確認して再試行してください  │
│                          [再試行]     │
└──────────────────────────────────────┘
```

### Level 4: ページ全体エラー（Error Boundary）

```
┌──────────────────────────────────────┐
│                                      │
│         ⚠️                           │
│                                      │
│    予期しないエラーが発生しました       │
│                                      │
│  問題が続く場合はサポートにご連絡ください │
│                                      │
│  [ページを再読み込み]  [トップへ戻る]   │
│                                      │
└──────────────────────────────────────┘
```

## Interaction Flow

### Level判定基準

```
エラーの影響範囲
  ├→ 単一フィールド             → Level 1（インライン）
  ├→ ページ内の操作失敗          → Level 2（Toast）
  ├→ セクション/コンポーネント    → Level 3（インラインバナー）
  └→ ページ全体のクラッシュ       → Level 4（Error Boundary）
```

### 回復フロー（Level 2・3）

```
エラー発生
  └→ エラー表示（原因 + 解決策）
      ├→ ユーザーが再試行
      │    └→ 成功 → エラー消去 + 成功Toast
      │    └→ 失敗 → エラー継続（試行回数カウント）
      ├→ 3回以上失敗
      │    └→ 「サポートに連絡」リンクを表示
      └→ ユーザーがキャンセル
           └→ エラー消去（操作前の状態に戻す）
```

## Error Message ガイドライン

### フォーマット

```
[何が起きたか] + [どうすればいいか]
```

### Good / Bad 例

| ❌ Bad | ✅ Good |
|--------|--------|
| 「エラーが発生しました」 | 「保存に失敗しました。通信環境を確認して再試行してください」 |
| 「入力が正しくありません」 | 「メールアドレスの形式が正しくありません（例: name@example.com）」 |
| 「403 Forbidden」 | 「この操作を行う権限がありません。管理者にお問い合わせください」 |
| 「必須項目です」 | 「名前を入力してください」 |

## Accessibility

- Level 1: `aria-describedby` でエラーメッセージをフィールドに紐付け + `aria-invalid="true"`
- Level 2: Toast に `role="alert"` + `aria-live="assertive"` → 即時読み上げ
- Level 3: バナーに `role="alert"` + `aria-live="polite"`
- Level 4: `<main>` 内の最初の要素としてエラーコンテンツを配置

## Do / Don't

### Do
- ✅ エラーメッセージは具体的に → ユーザーが原因と解決策を理解できる
- ✅ フォームエラーはサブミット後にフォーカスを最初のエラーフィールドに移動する
- ✅ 再試行可能なエラーには必ず「再試行」ボタンを提供する
- ✅ エラーログをSentry等に送信して開発者が把握できるようにする

### Don't
- ❌ 技術的なエラーコード（500, ECONNREFUSED等）をユーザーに見せない
- ❌ エラー後に入力済みデータを消去しない → 再入力の手間を最小化
- ❌ 全てのエラーにモーダルダイアログを使わない → インラインで十分な場合が多い
