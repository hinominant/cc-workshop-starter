# Implementation Checklist

> Builderの実装フロー全体を通すチェックリスト。各フェーズで抜け漏れを防ぐ。

---

## Pre-Implementation（実装前）

| # | チェック項目 | 確認方法 |
|---|------------|---------|
| 1 | 仕様/タスク記述を全文読んだ | 曖昧な箇所をリストアップできる |
| 2 | 曖昧な点を Nexus/Sherpa に確認済み | 推測で進めていない |
| 3 | 対象ディレクトリの既存コードを調査した | 命名規則・パターンを把握している |
| 4 | 既存テストのパターンを把握した | テストフレームワーク・スタイルが分かる |
| 5 | 変更対象ファイルが5つ以下 | 超える場合はスコープを再評価 |
| 6 | 新規依存パッケージの必要性を確認 | 既存ライブラリで代替できないか検討済み |
| 7 | Forge から引き継ぎの場合、`forge-insights.md` を読んだ | プロトタイプコードは無視、インサイトだけ抽出 |
| 8 | DB スキーマ変更が不要なことを確認 | 必要なら Schema エージェントに委譲 |

---

## During Implementation（実装中）

| # | チェック項目 | 判断基準 |
|---|------------|---------|
| 1 | テストを先に書いた | RED状態（失敗）を確認してから実装 |
| 2 | 各ステップの変更が50行以下 | 超える場合は分割 |
| 3 | `any` 型を使っていない | `unknown` + 型ガードで代替 |
| 4 | `@ts-ignore` / `as unknown as X` がない | 型エラーは根本解決 |
| 5 | エラーパスを先に書いた | Happy path より先にエラーハンドリング |
| 6 | 空の catch ブロックがない | 全 catch で意味のある処理 |
| 7 | ハードコード値がない | 設定値・定数に抽出 |
| 8 | `console.log` が残っていない | デバッグログは削除 |
| 9 | 既存の命名規則に合わせている | ファイル名・変数名・関数名 |
| 10 | import 順序が既存に合っている | プロジェクト規約に従う |
| 11 | 循環依存を導入していない | 依存方向が一方向 |

---

## Post-Implementation（実装後）

| # | チェック項目 | 確認方法 |
|---|------------|---------|
| 1 | 全テストが通る（既存 + 新規） | `npm test` / `vitest run` でゼロ失敗 |
| 2 | 新機能/変更に対するテストがある | 追加した関数・変更した振る舞い全てにテスト |
| 3 | 型チェックが通る | `tsc --noEmit` でゼロエラー |
| 4 | lint が通る | `npm run lint` でゼロ警告 |
| 5 | コンソールに警告がない | ランタイム警告もチェック |
| 6 | N+1 クエリがない | DB アクセスパターンを確認 |
| 7 | 不要な再レンダリングがない（React の場合） | React DevTools / StrictMode で確認 |
| 8 | エッジケースをテストした | null, 空配列, 境界値, 大量データ |
| 9 | コメントアウトされたコードがない | 不要コードは削除（git history に残る） |
| 10 | 仕様にない機能を追加していない | スコープクリープを防止 |

---

## Handoff Checklist（引き渡し）

### → Radar（テスト検証）

- [ ] 実装サマリを記述（何を作ったか）
- [ ] 期待する振る舞いをリスト化
- [ ] 既知のエッジケースを明記
- [ ] テスト実行コマンドを記載
- [ ] 環境依存の注意事項があれば記載

### → Judge（コードレビュー）

- [ ] Guardian と連携してコミットを整理済み
- [ ] 実装で行った判断とその理由を記述
- [ ] 代替案を検討した場合、なぜ採用しなかったかを記述
- [ ] S1/S2 レベルの問題がないことを自己確認

---

## Common Pitfalls（よくある落とし穴）

| シグナル | 問題 | 対処 |
|---------|------|------|
| ファイル変更が6つ以上 | スコープ肥大 | タスクを分割して段階実装 |
| テスト名が `should work` | 振る舞いが不明確 | テスト名を要件として書き直す |
| モックが5個以上 | 統合テストが必要 or 設計に問題 | Fake に置き換え or 設計を再検討 |
| `as` キャストが3箇所以上 | 型定義が不十分 | 型を修正してキャストを除去 |
| catch で `console.error` だけ | エラーが握り潰されている | リカバリ or リスロー or Result 型 |
| 同じ値を3箇所以上にハードコード | 定数化漏れ | 定数/設定に抽出 |
| テストが実行順序に依存 | テスト間結合 | 各テストを独立させる |
| 実装後にテストを書いている | TDD 違反 | RED → GREEN → REFACTOR に戻る |
| 仕様にない「便利機能」を追加 | スコープクリープ | 削除して Nexus に提案だけする |
| 既存パターンと違う書き方 | 一貫性の欠如 | 既存コードに合わせる |
