# Safe Refactoring Protocol

Zenがリファクタリング時に安全性を担保するためのプロトコル。行動を変えずにコードを改善するための手順と判断基準。

---

## Pre-Refactoring Checklist

リファクタリング開始前に全項目を確認する。1つでもNOなら対応してから着手。

```
[ ] テストが存在するか?
    → NO: Radar にカバレッジ追加を依頼。テストなしでリファクタしない
[ ] テストが全てグリーンか?
    → NO: 先にテストを修復。レッドの状態でリファクタしない
[ ] git working tree がクリーンか?
    → NO: 変更をコミットまたはスタッシュ。混在diffは事故の元
[ ] 対象コードの変更頻度は?
    → git log --oneline --since="3 months ago" -- <file> で確認
    → 変更0回なら優先度を下げる
[ ] リファクタリングの目的が明確か?
    → "きれいにする" は目的ではない。具体的な問題を特定する
[ ] 影響範囲を把握しているか?
    → grep で参照箇所を確認。パブリックAPIの変更は要承認
```

---

## Atomic Commit Strategy

**原則: 1リファクタリング = 1コミット**

各コミットは:
- 独立してリバート可能
- テストがグリーンのまま
- 50行以内のdiff
- コミットメッセージにリファクタリングの種類を明記

```
refactor: Extract calculateTax from processOrder
refactor: Replace nested conditional with guard clauses in validateInput
refactor: Introduce DateRange parameter object for report functions
```

**禁止:**
- リファクタリングとバグ修正を同一コミットに入れない
- リファクタリングと機能追加を同一コミットに入れない
- 複数の独立したリファクタリングを1コミットにまとめない

## Test-Driven Refactoring

### Red-Green-Refactor サイクル

```
1. GREEN を確認 (既存テストが全パス)
2. REFACTOR を実行 (コード構造を変更)
3. GREEN を確認 (全テストが変わらずパス)
4. 必要なら REFACTOR を継続 (次の小さなステップ)
5. GREEN を確認
6. COMMIT
```

### テスト実行タイミング

| タイミング | 実行するテスト | 目的 |
|-----------|---------------|------|
| リファクタ開始前 | 全テスト | ベースラインの確立 |
| 各ステップ後 | 対象ファイルのテスト | 高速フィードバック |
| コミット前 | 全テスト | リグレッション検出 |
| PR作成前 | 全テスト + lint | 最終確認 |

## Behavior Preservation Verification

リファクタリング後に動作が変わっていないことを確認する方法。

### 自動検証
1. **既存テストの全パス** - 最低条件
2. **スナップショットテストのdiff確認** - 意図しないUI/出力変更がないか
3. **型チェックのパス** - `tsc --noEmit` でコンパイルエラーなし
4. **lint のパス** - 新たな警告が出ていないか

### 手動検証 (テストでカバーできない場合)
1. **Before/After の動作比較** - 同じ入力で同じ出力が得られるか
2. **エッジケースの確認** - null, 空配列, 境界値での動作
3. **エラーハンドリングの確認** - 例外パスが変わっていないか
4. **副作用の確認** - ログ出力、DB書き込み、API呼び出しが同一か

### 危険シグナル (動作変更の可能性)
- `===` を `==` に変えた (またはその逆)
- 実行順序を変えた (副作用のある処理)
- デフォルト値を変えた
- エラーの型や内容を変えた
- 非同期処理の await を追加/削除した

## Rollback Strategy

### 即座にリバートすべきケース
- リファクタ後にテストが1つでも失敗
- リファクタ後に型エラーが発生
- リファクタの方向性が間違っていたと気づいた

### リバート手順
```bash
# 直前のコミットをリバート
git revert HEAD

# 特定のリファクタリングコミットだけリバート
git revert <commit-hash>

# コミット前なら
git checkout -- <file>
```

### リバートではなく修正すべきケース
- テストの期待値が古くて失敗している (テスト側を修正)
- 型定義の更新が必要 (型を修正)

**原則: 10分以内に直せないならリバート。デバッグにハマらない。**

## Legacy Code (テストなし) のリファクタリング

テストがないコードをリファクタリングする必要がある場合の段階的アプローチ。

### Step 1: Characterization Test (特性テスト)
現在の動作をそのまま記録するテストを書く。正しい動作ではなく、現在の動作を記録する。

```typescript
// 特性テスト: 現在の動作を記録
test("processOrder returns formatted string for valid order", () => {
  const order = { items: [{ name: "A", price: 100 }], tax: 0.1 };
  // 実際に実行して結果を見てから期待値を書く
  expect(processOrder(order)).toBe("Total: 110");
});
```

### Step 2: Seam (接合部) を見つける
テストしやすいように依存関係を注入できるポイントを特定する。

### Step 3: 安全な最小リファクタリング
- Rename Variable (最もリスクが低い)
- Extract Function (ロジックを変えない)
- Guard Clauses (条件の順序のみ変更)

### Step 4: テストを追加しながら段階的に改善
各ステップで特性テストを追加し、カバレッジを上げていく。

## When NOT to Refactor

リファクタリングを見送るべき状況。

| 状況 | 理由 |
|------|------|
| **Feature freeze 期間中** | リスクを増やすタイミングではない |
| **リリース直前** | 安定性が最優先 |
| **コードの理解が不十分** | 間違ったリファクタリングは元のコードより悪い |
| **テストがなく、追加する時間もない** | 安全ネットなしのリファクタは危険 |
| **近々リプレースが決まっている** | 投資対効果がない |
| **パフォーマンスクリティカルなコード** | 可読性のためにパフォーマンスを犠牲にしない (Bolt の領域) |
| **生成コード (migration, schema)** | 手で編集するものではない |
| **サードパーティ / ベンダーコード** | 所有権がないコードは変更しない |

### 判断フローチャート

```
リファクタリングしたい
├── テストがあるか? → NO → テスト追加の時間はあるか? → NO → 見送り
├── リリースまで1週間以内か? → YES → 見送り
├── コードの動作を完全に理解しているか? → NO → 先に理解を深める
├── 対象コードは今後変更される予定があるか? → NO → 優先度を下げる
└── 上記全てクリア → リファクタリング開始
```
