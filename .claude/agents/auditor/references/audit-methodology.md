# Audit Methodology

Auditorエージェントが品質監査を実施するための方法論。
プロセス準拠性、判断品質、リスク検知の3軸で監査を行う。

---

## 監査の原則

1. **監査は障害ではなく品質の守護者**: 早期検知で手戻りコストを最小化する
2. **エビデンスに基づく**: 推測で指摘しない。証拠を示す
3. **指摘だけで終わらない**: 必ず改善提案を添える
4. **SPECは法律**: 仕様書からの逸脱は例外なく REJECT

---

## 1. 仕様準拠監査プロセス

### Step 1: スコープ決定

監査対象の範囲を明確にする。

| 変更規模 | 監査スコープ |
|---------|------------|
| **Small** (1-3ファイル) | 該当ファイルの仕様準拠チェック |
| **Medium** (4-15ファイル) | 該当機能の仕様準拠 + 隣接機能への影響 |
| **Large** (16+ファイル) | 全体アーキテクチャ整合性 + 仕様準拠 + データフロー検証 |
| **Critical** (DB/認証/決済) | 全項目 + セキュリティ専門チェック + 本番安全性検証 |

### Step 2: 仕様の特定

```
監査対象の実装 → 対応する仕様書の特定 → 仕様書の要件一覧化
```

仕様書が存在しない場合は、監査を中断し仕様書作成を要求する。
「仕様書がない = 監査基準がない = 品質を保証できない」

### Step 3: ギャップ分析

```markdown
| 仕様要件 | 実装状態 | 判定 | 乖離内容 |
|---------|---------|------|---------|
| [要件1] | [実装内容] | OK/NG | [乖離があれば詳細] |
```

### Step 4: 発見事項の分類

| 分類 | 定義 | 対応期限 |
|------|------|---------|
| **Critical** | ユーザー安全性・データ整合性に直接影響。本番障害のリスク | 即時対応（デプロイブロック） |
| **Major** | 仕様からの明確な逸脱。機能が正しく動作しない | 当該PR/リリース前に修正必須 |
| **Minor** | 仕様の軽微な逸脱。機能は動作するが品質基準を満たさない | 次スプリント以内に対応 |
| **Observation** | 現時点では問題ないが、将来リスクになり得るパターン | 記録し定期監査で追跡 |

### Step 5: レポート作成

後述のレポートテンプレートに従い出力する。

## 2. 品質次元

監査で評価する5つの品質次元。

### 2.1 正確性（Correctness）

- 仕様通りの動作をするか
- 計算結果は正しいか（一次ソースとの突合）
- エッジケースで破綻しないか

**検証方法**: 仕様書の受入条件 → テストケース → 実行結果の三点突合

### 2.2 完全性（Completeness）

- 仕様の全要件が実装されているか
- 漏れている要件はないか
- エラーハンドリングは網羅されているか

**検証方法**: 仕様書の要件チェックリスト作成 → 実装との1:1マッピング

### 2.3 一貫性（Consistency）

- 同じ概念に複数の定義が存在しないか（SSoT原則）
- データフローの上流と下流で値が一致するか
- UI表示とDB値が一致するか

**検証方法**: データチェーン検証（Write→DB→Read全経路）

### 2.4 セキュリティ（Security）

- 認証・認可は適切か
- 入力バリデーションは行われているか
- 本番DBへの書き込みコードが存在しないか（READ ONLY原則）
- シークレットがコードに埋め込まれていないか

**検証方法**: Sentinelエージェントとの連携 + LROS本番安全性チェック

### 2.5 パフォーマンス（Performance）

- N+1クエリは存在しないか
- 大量データでの処理時間は許容範囲内か
- メモリ使用量は適切か

**検証方法**: Boltエージェントとの連携 + 負荷テスト結果の確認

## 3. 証拠収集基準

監査の指摘にはエビデンスが必須。以下の基準で収集する。

| 証拠タイプ | 収集方法 | 保存先 |
|-----------|---------|-------|
| **スクリーンショット** | 実画面の表示状態をキャプチャ | `artifacts/audit/` |
| **テスト結果** | テスト実行ログ（PASS/FAIL + エラー内容） | CI/CDログ or `artifacts/audit/` |
| **メトリクス** | カバレッジ率、レスポンスタイム、エラー率 | 計測ツールの出力 |
| **データ突合** | 一次ソース vs 加工後データの比較表 | `artifacts/audit/` |
| **git diff** | 変更内容の差分 | git history |
| **Activity Log** | エージェントの作業記録 | `.agents/PROJECT.md` |

**原則**: 一次ソースのみを真とする。LROS/NOVA/ARISの加工後データは正しさの根拠にならない。

## 5. 再監査トリガー

以下の場合、修正完了後に再監査を実施する。

| トリガー | 再監査スコープ |
|---------|--------------|
| Critical 発見事項の修正後 | 該当箇所 + 関連する依存箇所 |
| Major 発見事項の修正後 | 該当箇所のピンポイント確認 |
| アーキテクチャ変更後 | 全体整合性チェック |
| DB スキーマ変更後 | データフロー全体 |
| 認証・認可の変更後 | セキュリティ全項目 |

### 再監査プロセス

```
修正完了 → 修正内容のレビュー → 再監査実施 → 新たな問題がないか確認 → APPROVE/REJECT
```

**注意**: 修正のfix連鎖が2回連続した場合、パッチを重ねるのではなく設計自体を疑う（JC-TECH原則）。

## 6. 監査タイミングの判断基準

### 事前監査（Pre-Implementation）が必要なケース

- 新しいDB DDL作成
- API設計書のレビュー
- アーキテクチャ変更提案
- セキュリティに影響する設計変更

### 事後監査（Post-Implementation）が必要なケース

- 実装完了後、PR作成時
- テスト全通過後、デプロイ前
- リリース前の最終確認

### 定期監査（Periodic）

- 新フェーズ開始時: 前フェーズの完了確認
- 月次: 全体整合性チェック + Audit Warning追跡
- 四半期: 長期戦略前提の再検証

---

## 7. 監査品質の自己検証

監査自体の品質を担保するチェック。

- [ ] 全ての指摘にエビデンスがあるか
- [ ] 改善提案は具体的で実行可能か
- [ ] SPECに記載のない独自基準を追加していないか
- [ ] 推測で仕様を解釈していないか（不明点は質問として明示）
- [ ] 過去の失敗パターン辞書を参照したか
