# ARIS Audit Patterns

ARIS固有の監査パターン集。
ARISの判断プロセス・データ品質・意思決定の追跡可能性を検証する。

---

## 1. Four Minds 一貫性チェック

### 目的

CEOの判断がFour Minds全てを通過しているか、特定のMindがスキップされていないかを検証する。

### チェック項目

| # | 検証内容 | 検出パターン | 判定 |
|---|---------|------------|------|
| 1 | 4つのMind全てが評価を出しているか | Mindの評価が1つでも欠落 | REJECT |
| 2 | Founderが最初に評価されているか | Vision/Executionが先行 | WARN |
| 3 | Auditが全判断を監査しているか | Audit評価なし | REJECT |
| 4 | Mind間の矛盾が解消されているか | 矛盾が放置されたまま結論 | REJECT |
| 5 | Founderの拒否が尊重されているか | Founder FAIL + 全体 Go | CRITICAL REJECT |

### 監査手順

```
1. .context/aris-decisions.jsonl を読み込む
2. 各判断レコードの minds フィールドを検証
3. founder: "FAIL" かつ decision: "Go" のレコードを検出 → CRITICAL
4. minds に null/空文字のフィールドがあるレコードを検出 → REJECT
5. audit: 未記入のレコードを検出 → REJECT
```

### 典型的な違反パターン

- **Founder Skip**: 「技術的に優れているからGo」-- Founder評価なしで判断
- **Audit Silence**: 全MindがGoを出した後、Auditが形式的PASSを出す
- **Vision Override**: Visionの成長性評価がFounderの思想チェックを上書き

## 2. Founder Core 整合性検証

### 目的

判断がFounder Core（不変原則）に整合しているかを検証する。

### 不変原則チェックリスト

| # | 原則 | 違反の兆候 |
|---|------|-----------|
| 1 | 売上は目的ではない。結果である | 「売上のために」が判断理由の先頭にある |
| 2 | 信念と文化が先 | 文化的影響の検討なしに効率性だけで判断 |
| 3 | 無理に広げない | 「市場シェア拡大のために」が主目的 |
| 4 | 否定されない自分らしさを守る | 外部圧力（競合・投資家）による方針転換 |
| 5 | 社長時間効率は最上位KPI | Keijiの手動作業が増える設計 |

### 最上位優先順位の遵守検証

```
判断の中で以下の順番が守られているか:
  法的安全 > 身体的安全 > 心理的安全 > 治安 > 尊厳 > 成長 > 利便性

違反例: 「利便性のために心理的安全を少し犠牲にする」 → REJECT
```

## 3. KPI整合性検証

### 目的

KPIの「定義→計算→表示」チェーンが一貫しているかを検証する。

### データチェーン検証

```
一次ソース（Luna DB）
    ↓ 取り込み
raw レイヤー（LROS）
    ↓ 加工
mart レイヤー（fact/dim）
    ↓ 集計
metrics レイヤー
    ↓ 表示
ダッシュボード/レポート
```

### チェック項目

| # | 検証内容 | 方法 |
|---|---------|------|
| 1 | 指標定義の一意性 | 同じ指標が複数箇所で異なる定義を持っていないか |
| 2 | 計算ロジックの正確性 | LROS ABSOLUTE SPECの定義と実装の突合 |
| 3 | 一次ソースとの整合 | raw → mart → metrics の各段階で値が一致するか |
| 4 | 表示値の正確性 | metricsの値とダッシュボード表示値の突合 |
| 5 | 定義変更の追跡 | 指標定義が変更された場合、変更ログがあるか |

### SSoT原則

- 全指標の唯一の定義元は LROS の `metrics_monthly`
- 独自定義の作成は禁止
- 一次ソース（Luna DB直接/元CSV）のみが正しさの根拠
- LROS/NOVA/ARISの加工後データは正しさの根拠にならない

### 典型的な違反パターン

- **定義の二重管理**: LROSの定義とは別にスプレッドシートで独自定義
- **計算ロジックのドリフト**: SPEC更新時に実装が追従していない
- **サイレント精度劣化**: 取り込み失敗が通知されず、古いデータで計算が継続

## 4. 意思決定監査証跡

### 目的

判断が記録され、後から追跡可能であることを検証する。

### 必須記録項目

| 項目 | 必須 | 不在時の判定 |
|------|------|------------|
| 判断日 | 必須 | REJECT |
| 判断対象 | 必須 | REJECT |
| 判定結果（Go/No-Go） | 必須 | REJECT |
| Four Minds評価 | 必須 | REJECT |
| 根拠データ | 必須 | REJECT |
| 代替案（2案以上） | 必須 | MAJOR |
| 制約条件 | 条件付きGoの場合必須 | MAJOR |
| エスカレーション記録 | 該当時必須 | MAJOR |

### 監査証跡の検証手順

```
1. .context/aris-decisions.jsonl の全レコードを読み込む
2. 必須フィールドの欠落を検出
3. 条件付きGoの制約条件追跡（期限、達成状況）
4. エスカレーション記録の存在確認（該当ケース）
5. Activity Log との整合性確認
```

## 5. 委譲フォローアップチェック

### 目的

「委譲 = 完了」の誤認を防ぐ。委譲されたタスクの完了確認が行われているかを検証する。

### チェック項目

| # | 検証内容 | 検出パターン |
|---|---------|------------|
| 1 | 委譲時に完了基準が明示されているか | 「お願いします」だけで完了基準なし |
| 2 | 完了確認のタイミングが設定されているか | 確認日未設定 |
| 3 | 完了報告が記録されているか | Activity Logに完了記録なし |
| 4 | 委譲先の作業品質が検証されているか | 「やりました」の報告のみで成果物未確認 |

### 監査手順

```
1. Activity Log から委譲記録を抽出
2. 各委譲に対する完了記録の有無を確認
3. 完了基準が明示されているか確認
4. 2週間以上フォローアップなしの委譲を検出 → WARN
```

## 6. ARIS固有の監査発見パターン

過去の監査で頻出する発見事項のパターン。

### パターン A: 承認率の偏り

- **発見**: チーム提案の承認率が98.4%
- **リスク**: 質的フィルタリングの形骸化
- **対処**: 承認率の定期モニタリング + 却下理由の構造化

### パターン B: マーケティング判断の浅さ

- **発見**: 技術・プロダクトには四人格分析、マーケティングには感覚判断
- **リスク**: マーケティング投資のROI未検証
- **対処**: マーケティング施策にも同一の判断基準を適用

### パターン C: 採用判断のARIS未通過

- **発見**: 最大カテゴリ（41%）だが四人格セッション0件
- **リスク**: 組織の最重要判断が構造化されていない
- **対処**: 採用判断にもFour Mindsプロトコルを適用

### パターン D: 条件付きGoの漂流

- **発見**: 条件付き承認に期限がなく、条件未達のまま進行
- **リスク**: 中途半端な状態の長期化
- **対処**: 全条件付きGoに2週間期限を設定

### パターン E: Audit警告の「認識済み」放置

- **発見**: 警告を「認識した」が対策未実行
- **リスク**: 認識されたリスクが顕在化
- **対処**: 警告ステータスの4段階追跡（認識→立案→実行→効果確認）

## 7. エスカレーションパス

### Critical 発見時

```
Auditor → CEO（即時）→ Keiji（不可逆な判断の場合）
```

### Major 発見時

```
Auditor → Nexus → 担当エージェント（修正依頼）→ Auditor（再監査）
```

### WARN 発見時

```
Auditor → Activity Log に記録 → 次回定期監査で追跡
```

### 監査結果のフィードバック

- 成功パターン: `/log-success` でARIS成功パターン辞書に登録
- 失敗パターン: `/log-failure` でARIS失敗パターン辞書に登録
- 重複チェック必須: 同カテゴリ既存パターンを全件確認してから追加
