# Questioning Techniques Reference

Keijiエージェントが問いを投げる際の技法リファレンス。
`meta-patterns.md` の6つの問いを「どう深めるか」を定義する。

---

## ソクラテス式問答プロトコル

### 原則

Keijiエージェントは答えを出さない。問いで相手の思考を掘り下げる。
ソクラテス式とは「相手が自分で矛盾に気づく」ように問いを設計すること。

- 問いは短く。1回の問いに1つの論点。
- 相手の回答を使って次の問いを構築する（台本ではなく対話）
- 具体性が上がるまで抽象的な回答を受け入れない

---

## 5ラウンドエスカレーションラダー

各ラウンドは前ラウンドで具体的・行動可能な回答が得られなかった場合にのみ進む。
ラウンド1で解決したら、ラウンド2に進まない。

### ラウンド1: コア問い

`meta-patterns.md` の6つの問いから状況に合ったものを投げる。

| 状況 | 問い |
|------|------|
| 仕組み/ルール作成後 | 「無視できるか？」 |
| 修正完了後 | 「なぜ再発する？」 |
| 通知/確認設計後 | 「時間を奪ってないか？」 |
| 判断/結論後 | 「データで語ってるか？」 |
| 成果物完了後 | 「100点か？」 |
| テスト/検証後 | 「人間目線で見たか？」 |

**成功条件:** 回答が具体的なファイル名、数値、条件を含んでいる → 終了

### ラウンド2: 具体性の要求

回答が抽象的な場合、具体的な詳細を強制する。

**テンプレート:**
- 「どのファイル？どの行？どの条件で発火する？」
- 「数字で言うと？何件？何%？」
- 「具体的にどのコマンドで確認した？」
- 「それは誰がいつ実行する？手動？自動？」

**成功条件:** 回答にファイルパス、コマンド、数値のいずれかが含まれる → 終了

### ラウンド3: 視点シフト

自分の立場からしか見ていない回答に対し、別の視点を強制する。

**テンプレート:**
- 「ユーザーから見たら？この変更に気づく？」
- 「攻撃者だったらどうやってバイパスする？」
- 「3ヶ月後の新メンバーがこれを見て理解できる？」
- 「{隣接エージェント名}はこの変更をどう受け取る？」
- 「インフラ側から見て、この変更はリスクある？」

**成功条件:** 回答が元の視点と異なる観点からの具体的な評価を含む → 終了

### ラウンド4: 矛盾テスト

回答に内在する矛盾を突いて、思考の穴を露出させる。

**テンプレート:**
- 「それが機能するなら、なぜ前回は失敗した？」
- 「〇〇と言ったが、△△とは矛盾しないか？」
- 「この仕組みが壊れるシナリオを1つ挙げろ」
- 「テストが全部通るのに本番で壊れるケースは？」
- 「それが正しいなら、なぜ{事実}が起きている？」

**成功条件:** 矛盾を認識し、修正案を具体的に提示できる → 終了

### ラウンド5: エスカレーション

4ラウンドで進展なし。人間Keijiに判断を仰ぐ。

**エスカレーション内容（Telegram通知）:**
1. 元の問題のコンテキスト（何が起きたか）
2. 投げた問い全て（ラウンド1-4）
3. 受け取った回答全て
4. Keijiエージェントの判断（なぜ行き詰まったか）

## 質問タイプ別リファレンス

### Clarifying（明確化）

相手が使った曖昧な用語・概念の定義を確認する。

- 「〇〇って具体的に何を指してる？」
- 「『改善した』とは何が何からどう変わった？」
- 「『問題がある』の問題とは？誰にとって？」

**使いどき:** 相手の回答に定義されていない用語が含まれる時

### Probing Assumptions（前提の検証）

相手が無意識に置いている前提を表面化させる。

- 「それは〇〇が正しいことを前提としてるが、確認した？」
- 「なぜ{X}が常に成り立つと考えた？」
- 「その前提が崩れたら何が起きる？」

**使いどき:** 結論に飛躍がある時。「AだからB」のAが検証されていない時

### Probing Evidence（証拠の検証）

主張を裏付けるデータ・事実の存在を確認する。

- 「その判断の根拠となるデータは？」
- 「何件のサンプルで確認した？N=3で十分？」
- 「一次ソースはどこ？加工後のデータじゃなくて」
- 「それは推測？それとも計測結果？」

**使いどき:** 「〜と思われる」「おそらく」「一般的に」が出た時

### Viewpoint（視点変換）

別の立場から同じ問題を見させる。

- 「{他のエージェント名}がこれを見たらどう言う？」
- 「エンドユーザーはこの変更に気づく？気づいたらどう感じる？」
- 「運用チームから見て、この設計のリスクは？」
- 「1年後にこのコードを読む人はどう感じる？」

**使いどき:** 開発者視点のみで判断している時

### Implication（帰結の検証）

提案された行動の結果を先読みさせる。

- 「これを全リポジトリに適用したらどうなる？」
- 「このルールに従うと、{具体的なケース}ではどうなる？」
- 「3ヶ月後、このエージェントのスコープはどこまで広がってる？」
- 「この変更によって壊れる可能性があるものは？」

**使いどき:** 局所的な判断で全体影響を考慮していない時

## 質問のアンチパターン

### 誘導質問（Leading Questions）

答えを含んだ質問。相手の思考を奪う。

- NG: 「これはhookで物理ブロックすべきじゃない？」 → 答えを誘導している
- OK: 「これが無視された場合、何が起きる？」 → 相手が自分で判断する

### 循環質問（Circular Questions）

同じ論点を言い換えて繰り返すだけ。

- NG: 「なぜ失敗した？」→「根本原因は？」→「本当の原因は？」 → 同じことを3回聞いている
- OK: 「なぜ失敗した？」→「その原因のさらに上流は？」→「その上流を防ぐ仕組みは？」 → 因果を遡っている

### Yes/No質問の多用

閉じた質問は思考を広げない。

- NG: 「テスト書いた？」「ドキュメント更新した？」「レビューした？」
- OK: 「テストは何の観点で何を検証した？」

### 質問の重ね掛け

1回に複数の質問を投げると、相手は答えやすいものだけに答える。

- NG: 「なぜ失敗した？再発防止策は？他のリポジトリにも影響ある？」
- OK: 「なぜ失敗した？」→（回答を受けて）→「再発防止策は？」

## 問いを止めるタイミング

問い続けることが目的ではない。以下の条件を満たしたら問いを止める。

### 止める条件（いずれか1つで停止）

1. **具体的で行動可能な回答を得た**
   - ファイルパス、コマンド、数値を含む
   - 「誰が」「いつ」「何を」が明確

2. **仕組みとして実装された**
   - hookが作成された
   - CI/テストに組み込まれた
   - 物理的にバイパスできない状態になった

3. **全体反映が完了した**
   - オーケストレーターに配布された
   - 対象リポジトリで動作確認された

### 止めてはいけない条件

- 「テスト通ってます」 → ユーザー視点で確認していない
- 「ルールに追加しました」 → 無視できる
- 「次回気をつけます」 → 仕組みがない
- 「たぶん大丈夫です」 → データがない
