# 技術的プライバシーパターン — datashield（副査）

> 最終更新: 2026-04-06
> 対象: datashield エージェント
> 用途: privacyの評価を技術的実効性の視点で検証するためのパターン集

---

## 1. 暗号化パターン

### 1.1 保存時暗号化（Encryption at Rest）

| パターン | 実装レベル | 保護対象 | 検証方法 |
|---------|-----------|---------|---------|
| ディスク暗号化 | インフラ | ストレージ全体 | AWS: EBS暗号化設定確認、GCP: CMEK設定確認 |
| データベース暗号化 | DB | テーブル/カラム | TDE設定確認、暗号化カラムの設定確認 |
| フィールドレベル暗号化 | アプリ | 特定フィールド | PII カラムの暗号化実装コード確認 |
| エンベロープ暗号化 | アプリ | データキー自体 | KMSでマスターキー管理、データキーをローテーション |

#### 検証チェックリスト

- [ ] 暗号化アルゴリズムが適切か（AES-256-GCM推奨。DES/3DES/RC4は不可）
- [ ] 暗号化キーがデータと同一ストレージに保存されていないか
- [ ] キーローテーションが設定されているか（90日〜1年推奨）
- [ ] バックアップデータも暗号化されているか
- [ ] 一時ファイル・キャッシュも暗号化対象に含まれているか
- [ ] ログに個人情報が平文で出力されていないか

### 1.2 通信暗号化（Encryption in Transit）

| パターン | 適用範囲 | 最低要件 |
|---------|---------|---------|
| TLS | クライアント ↔ サーバー | TLS 1.2以上。TLS 1.0/1.1は無効化必須 |
| mTLS | サービス ↔ サービス | 内部通信でもサービスメッシュ等でmTLSを適用 |
| VPN/WireGuard | 拠点間通信 | 暗号化トンネルで全通信を保護 |
| SSH | 運用アクセス | 公開鍵認証必須。パスワード認証は禁止 |

#### 検証チェックリスト

- [ ] 全外部通信がTLS 1.2以上で暗号化されているか
- [ ] 内部のマイクロサービス間通信が暗号化されているか（平文HTTP禁止）
- [ ] TLS証明書の有効期限が管理されているか（自動更新推奨）
- [ ] 脆弱な暗号スイート（RC4, DES, 3DES, MD5）が無効化されているか
- [ ] HSTS（HTTP Strict Transport Security）が設定されているか

### 1.3 エンドツーエンド暗号化（E2E Encryption）

| 適用場面 | 実装パターン | 検証ポイント |
|---------|------------|-------------|
| メッセージング | クライアントサイドで暗号化、受信者のみ復号 | サーバー側で平文にアクセスできないことの確認 |
| ファイル共有 | アップロード前にクライアントで暗号化 | サーバー上のファイルが暗号化状態であることの確認 |
| AI API連携 | 送信前にPIIをマスクし、サーバー側で平文PIIに触れない | プロンプト内のPIIマスキング実装の確認 |

## 2. アクセス制御パターン

### 2.1 PII向けRBAC（Role-Based Access Control）

| ロール | アクセス権限 | 対象データ |
|--------|-----------|-----------|
| admin | フルアクセス（読み書き削除） | 全個人データ。ただし操作ログ必須 |
| data_operator | 業務に必要な範囲の読み書き | 担当業務のデータのみ |
| analyst | 匿名化/仮名加工データの読み取り | 非識別データのみ |
| auditor | 読み取り専用（監査証跡含む） | 全データ（ただしコピー不可） |
| service_account | API経由の自動処理 | 定義されたスコープのみ |
| external_partner | 限定的な読み取り | 共有対象として合意されたデータのみ |

#### 実装要件

- [ ] RBACが実装され、デフォルトdenyの設計か
- [ ] ワイルドカード権限（`*:*`）が本番環境に存在しないか
- [ ] ロール定義と実際のアクセス権限が一致しているか
- [ ] ロールの割り当てに承認プロセスがあるか

### 2.2 フィールドレベルアクセス制御

| パターン | 説明 | 実装例 |
|---------|------|--------|
| Column-level ACL | カラムごとにアクセス権限を制御 | PostgreSQL: GRANT SELECT (name, email) ON users TO analyst_role |
| Dynamic Masking | ロールに応じてデータをマスクして返却 | SQL Server Dynamic Data Masking、アプリレイヤーでのマスキングミドルウェア |
| View-based Control | ロール別のビューを提供 | `users_public_view` (id, display_name) vs `users_admin_view` (全カラム) |
| API Field Filtering | APIレスポンスからロールに応じてフィールドを除外 | GraphQLのfield-level authorization、REST APIのfield filtering middleware |

## 3. 監査ログパターン

### 3.1 個人データアクセスの監査ログ要件

| ログ項目 | 内容 | 必須/推奨 |
|---------|------|----------|
| who | アクセスしたユーザー/サービスの識別子 | 必須 |
| what | アクセスしたデータの種類（テーブル名、カラム名） | 必須 |
| when | アクセス日時（UTC、ミリ秒精度） | 必須 |
| where | アクセス元のIPアドレス/サービス名 | 必須 |
| why | アクセス目的（API呼び出し元のpurposeヘッダー等） | 推奨 |
| how | アクセス方法（SQL直接/API/管理画面） | 推奨 |
| result | アクセス結果（成功/失敗/件数） | 必須 |

### 3.2 ログの保護パターン

| パターン | 説明 | 実装 |
|---------|------|------|
| Append-only Storage | ログの上書き・削除を物理的に不可能にする | S3 Object Lock、WORM ストレージ |
| Hash Chain | ログエントリにハッシュチェーンを適用し改ざんを検知 | 各エントリに前エントリのハッシュを含む |
| Separate Account | ログを別アカウント/別テナントに保管 | AWS Organizations の別アカウントにCloudTrailログを集約 |
| Tamper Detection | ログの改ざんを自動検知しアラート | CloudTrail Log File Validation |

### 3.3 異常アクセス検知

| 検知パターン | 検知方法 | アクション |
|-------------|---------|-----------|
| 大量アクセス | 閾値を超えるデータ読み取り（例: 100件/分） | アラート + 自動ブロック |
| 時間外アクセス | 深夜・休日の個人データアクセス | アラート + 確認要求 |
| 権限外アクセス試行 | 403エラーの頻発 | アラート + アカウント一時ロック |
| 一括エクスポート | データの大量ダウンロード | 承認要求 + ログ強化 |
| 不明なソース | 未登録IPからのアクセス | ブロック + アラート |

## 4. データマスキング/トークン化パターン

### 4.1 マスキング手法

| 手法 | 用途 | 例 |
|------|------|-----|
| 部分マスク | 表示時の個人情報保護 | `田中***` / `tanaka@***.com` / `080-****-5678` |
| 完全マスク | 非権限者への非表示 | `***` |
| シャッフル | テスト環境でのデータ生成 | 姓と名をランダムに再組み合わせ |
| 置換 | テスト環境でのダミーデータ | Faker等でリアルなダミーに置換 |
| ノイズ付加 | 統計分析時のプライバシー保護 | 差分プライバシー（ε-DP） |

### 4.2 トークン化

```
[トークン化のアーキテクチャ]

Original PII → Token Vault (暗号化保管) → Token (非可逆参照)

                ┌─────────────┐
 "田中太郎" →  │  Token Vault │ → "tok_a1b2c3d4"
                │  (HSM保護)   │
                └─────────────┘
                      ↑
              復元は Token Vault 経由のみ
              アクセス制御 + 監査ログ必須
```

| 要件 | 基準 |
|------|------|
| トークン形式 | 元データと同じフォーマット（Format-Preserving Tokenization）が理想 |
| Vault保護 | HSM又は暗号化キー管理サービス（KMS）で保護 |
| 復元権限 | 復元可能なロールを最小限に制限 |
| 監査 | 全復元操作をログに記録 |

## 5. 安全な削除パターン

### 5.1 削除レベルの定義

| レベル | 方法 | 復元可能性 | 適用場面 |
|--------|------|-----------|---------|
| L1: 論理削除 | `deleted_at` フラグ | 完全に復元可能 | 誤削除対策の一時措置。最終削除ではない |
| L2: 物理削除 | `DELETE FROM` + VACUUM | DBレベルで復元困難 | 通常の削除処理 |
| L3: 暗号学的削除 | 暗号化キーの破棄 | データは残存するが復号不可能 | 大量データの効率的な削除 |
| L4: 完全消去 | ストレージレベルでの上書き消去 | 復元不可能 | 高機密データの廃棄 |

### 5.2 ソフトデリート（論理削除）の注意点

**論理削除だけでは「削除」とは認められない場面がある。**

| リスク | 内容 | 対策 |
|--------|------|------|
| 検索可能 | `WHERE deleted_at IS NOT NULL` で復元可能 | 論理削除後に一定期間（30日等）で物理削除を自動実行 |
| バックアップに残存 | バックアップから復元される | バックアップの保持期間を考慮した完全削除スケジュール |
| 関連データの残存 | 親レコードは削除したが子レコードが残っている | CASCADE DELETE の設計、関連テーブルの洗い出し |
| レプリカの残存 | マスターから削除したがレプリカに残っている | レプリケーション遅延を考慮した削除確認 |

### 5.3 削除の実装チェックリスト

- [ ] 削除要求から物理削除完了までのフローが定義されているか
- [ ] 論理削除→物理削除の自動スケジュールが設定されているか
- [ ] バックアップからの復元時に削除済みデータが復活しない仕組みがあるか
- [ ] 関連テーブル（外部キー参照先）のデータも同時に削除されるか
- [ ] 削除完了の記録（何を、いつ、どの方法で削除したか）が保持されているか
- [ ] 検索インデックス（Elasticsearch等）からも削除されているか
- [ ] CDN/キャッシュからも削除されているか

## 6. API セキュリティパターン（個人データエンドポイント）

### 6.1 認証・認可

| パターン | 説明 | 実装 |
|---------|------|------|
| OAuth 2.0 + PKCE | トークンベースの認可 | 個人データAPIには必ずスコープベースの認可を適用 |
| API Key + Rate Limit | 機械間通信の認証 | APIキーの定期ローテーション + レート制限 |
| JWT Claims Validation | トークン内のクレームで権限を検証 | `scope`, `sub`, `purpose` クレームの検証 |
| Field-level Authorization | フィールドレベルの認可 | GraphQL directive `@auth(requires: ADMIN)` |

### 6.2 データ漏洩防止

| パターン | 説明 | 実装 |
|---------|------|------|
| Response Filtering | 権限に応じてレスポンスフィールドを除外 | ミドルウェアで不要フィールドを除去 |
| Pagination Enforcement | 大量データの一括取得を防止 | `max_limit=100`, cursor-based pagination |
| Export Approval | データエクスポートに承認を要求 | 一定件数以上のエクスポートは承認ワークフロー経由 |
| Watermarking | エクスポートデータに利用者の識別情報を埋め込み | CSVヘッダーにエクスポート者IDを記録 |

### 6.3 APIセキュリティチェックリスト

- [ ] 個人データを返すエンドポイントに認証が必須か
- [ ] レスポンスに不要な個人情報が含まれていないか（over-fetching防止）
- [ ] エラーレスポンスに個人情報が含まれていないか
- [ ] ページネーションが強制されているか
- [ ] レート制限が設定されているか
- [ ] CORS設定が適切か（信頼されたオリジンのみ許可）
- [ ] APIバージョニングで古いエンドポイントが無効化されているか

## 7. データベースレベルのプライバシー制御

### 7.1 Row-Level Security（RLS）

| DB | 実装方法 | 用途 |
|----|---------|------|
| PostgreSQL | `CREATE POLICY` + `ALTER TABLE ENABLE ROW LEVEL SECURITY` | テナント分離、ユーザーデータの分離 |
| Supabase | RLS + `auth.uid()` | 自分のデータのみアクセス可能にする |
| MySQL | ビュー + `CURRENT_USER()` | 簡易的な行レベル制御 |

```sql
-- PostgreSQL RLS例: ユーザーは自分のデータのみ参照可能
CREATE POLICY user_data_isolation ON personal_data
  USING (user_id = current_setting('app.current_user_id')::uuid);
```

### 7.2 カラムレベル暗号化

| パターン | 実装 | トレードオフ |
|---------|------|------------|
| Application-level | アプリで暗号化してからDBに保存 | 検索不可（暗号化カラムのWHERE句不可） |
| DB-level TDE | DB機能で透過的に暗号化 | DBアクセス権があれば復号される |
| Column Encryption | 特定カラムのみ暗号化 | 暗号化カラムのインデックス作成に制約 |
| Searchable Encryption | ブラインドインデックスで暗号化カラムの検索を可能に | 実装複雑度が高い |

### 7.3 データベースセキュリティチェックリスト

- [ ] RLSが有効化され、テナント/ユーザー間のデータ分離が実装されているか
- [ ] PIIカラム（氏名、メール、電話番号、住所等）が暗号化されているか
- [ ] DBへの直接アクセス（psql/mysql CLI）が制限されているか
- [ ] DBユーザーの権限が最小限に設定されているか（SELECT ONLYの分析用ユーザー等）
- [ ] DB接続がTLS暗号化されているか（`sslmode=require`）
- [ ] 監査ログ（pgAudit等）が有効化されているか
- [ ] 自動バキューム/パージで削除データの物理的な痕跡が消去されるか
